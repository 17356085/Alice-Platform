|--------|----------|------|----------|------|---------------|------|
| BUSK-SRQ-001 | **申请人信息不实导致库存异常占用** | P0 | 申请人填写虚假名称或未授权人员提交领用申请 | 备品被冒领，库存数据不准确 | 1. 申请人身份校验<br>2. 审批链审核申请人资质<br>3. 使用后记录追踪 | 核心业务风险 |

# 业务场景模型: 备品领用申请 (Spare Requisition)

## 1. 业务目标

- **核心业务目标**: 提供备品领用的申请、审批、发放全流程管理，确保备品按需发放且有据可查。
- **次要目标**: 通过工作流状态按钮显隐机制，引导用户完成正确操作步骤。

## 2. 角色与旅程

| 角色名 | 职责边界 | 典型旅程 |
|--------|----------|----------|
| **申请人** | 创建并提交领用申请 | 新建申请 -> 填写申请人 -> 保存 -> 提交审批 -> 等待审批结果 |
| **审批人 (admin/chenqian)** | 第一级审批 | 查看待审批申请 -> 审批通过 -> 转入 tjw 审批 |
| **审批人 (tjw)** | 第二级审批 | 查看待审批申请 -> 最终审批通过/驳回 |
| **仓库管理员** | 执行备品发放 | 查看已审批申请 -> 发放备品 -> 完成流程 |
| **系统管理员** | 配置审批链和用户权限 | 配置备件领用申请审批链 |

## 3. 业务流程

### Happy Path
1. 申请人导航到备品领用申请页面
2. 点击新增，填写申请人信息
3. 保存 -> 记录出现在列表中，状态为"草稿/待提交"
4. 点击提交按钮 -> Toast 提示提交成功，状态变更
5. admin + chenqian 审批通过
6. tjw 审批通过
7. 仓库管理员发放备品

### Alternative Path
1. **取消申请**: 填写后取消，不保存申请
2. **编辑草稿**: 在草稿状态编辑申请内容
3. **删除草稿**: 删除草稿状态的申请
4. **审批驳回**: 审批人驳回申请 -> 申请人可编辑后重新提交
5. **查看详情**: 查看已提交申请的详细信息

### Exception Path
1. **必填校验失败**: 不填写申请人直接保存
2. **已审批记录不可删除**: 删除按钮隐藏或删除调用被 try/except 捕获
3. **重复提交**: 用户快速双击提交 -> 后端幂等处理
4. **审批链异常**: admin+chenqian 审批后 tjw 不可用

## 4. 业务规则

| 规则ID | 规则描述 | 校验阶段 | 自信度 |
|--------|----------|----------|--------|
| RULE-SRQ-001 | 编辑/提交/删除按钮仅草稿/待提交状态可见 | 前端 v-if | 中，从 PO 代码推断 |
| RULE-SRQ-002 | 已审批记录不可删除 | 后端接口 | 高，从 delete_by_name try/except 推断 |
| RULE-SRQ-003 | 提交后触发 Toast 提示而非弹窗确认 | 前端操作 | 高，从 click_submit_first 方法体 |
| RULE-SRQ-004 | 删除需 MessageBox 确认 + Toast 提示 | 前端操作 | 高，从 click_delete_first 方法体 |
| RULE-SRQ-005 | 审批链为两阶段：admin+chenqian 会签 -> tjw 审批 | 后端工作流 | 高，PO 文档字符串 |

## 5. 数据流

| 数据项 | 数据来源 | 数据变换 | 数据消费 |
|--------|----------|----------|----------|
| 申请人 | 用户填写 | 搜索条件/申请记录 | 列表展示/搜索筛选 |
| 流程状态 | 工作流引擎 | 草稿 -> 审批中 -> 已通过/已驳回 | 列表 el-tag 显示/按钮显隐控制 |
| 操作按钮 | 根据状态计算 | 按钮 v-if 条件渲染 | 用户交互入口 |
