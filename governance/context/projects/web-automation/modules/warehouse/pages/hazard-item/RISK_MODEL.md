## 1. 业务目标 (Business Goal)

-   **核心业务目标**：管理企业生产经营活动中产生的所有危险废物的种类和基本信息，建立一个统一的危废品目录，为后续的入库、存储、处置和上报提供基础数据支撑。
-   **次要业务目标**：支持快速查找、新增和清理危废品，保证危废品数据维护的效率和准确性。
-   **非目标**：本页面不处理危废品的入库、出库、处置等业务流程；不包含审批流程；不负责生成环保部门要求的报表。

## 2. 角色与旅程 (Roles & Journeys)

### 角色列表

| 角色 | 职责边界 | 典型操作 |
|---|---|---|
| **库管员** (库存管理员) | 日常维护危废品目录，确保目录准确可用。 | 查看、搜索、新增、删除 |
| **环保专员** | 审核和管理危废品目录，关注合规性。 | 查看、搜索、新增 |

### 典型业务旅程

**库管员 - “新增一种危废品” 旅程**
1.  **进入**：导航到 `库管管理 > 环保危废管理 > 物品管理`，页面成功加载并展示当前危废品列表。
2.  **确认**：通过搜索检查系统中是否已存在同名称的危废品。如果存在，则复用或标记，流程结束。如果不存在，继续下一步。
3.  **新建**：点击“新增”按钮，弹出新增表单。
4.  **填写**：在弹窗中填写危废品名称（AUTO_废油_001）。
5.  **保存**：点击“保存”按钮，表单提交，弹窗关闭。
6.  **验证**：页面自动刷新或用户手动刷新搜索，在列表中确认新危废品已出现。旅程结束。

**库管员 - “删除一种废弃危废品” 旅程**
1.  **寻找**：进入页面，通过搜索功能找到需要删除的危废品。
2.  **触发删除**：在对应的数据行点击“删除”按钮。
3.  **二次确认**：系统弹出确认弹窗（`confirm_message_box`），询问“确认删除 [危废品名] 吗？”。
4.  **确认执行**：点击确认弹窗中的“确定”按钮，删除请求发送至后端。
5.  **验证**：系统提示删除成功，列表刷新，被删除的危废品消失。旅程结束。

## 3. 业务流程 (Workflows)

-   **Happy Path (主流程)**：危废品名录维护流程 (本页面是起点)
    -   [库管员] 查看危废品目录 → 发现需要新增危废品 → 执行新增旅程 → 新增成功 → 目录更新。
    -   [库管员] 查看危废品目录 → 发现废弃危废品 → 执行删除旅程 → 删除成功 → 目录更新。
-   **Alternative Path 1 (取消新增)**：
    -   [库管员] 执行新增之旅，填写部分信息 → 在保存之前，点击弹窗“取消”按钮 → 弹窗关闭，页面无变化，未新增数据。
-   **Alternative Path 2 (重复新增)**：
    -   [库管员] 执行新增之旅，填写已存在的危废品名称 → 点击“保存” → 后端返回“危废品名称已存在”错误 → 前端显示错误消息，弹窗未关闭，保存失败。

## 5. 数据流 (Data Flow)

-   **数据来源**：
    -   本页面的危废品列表数据由 **`GET /api/warehouse/hazard-items`** 接口提供。
    -   本页面的数据由用户通过**新增弹窗**创建，请求**`POST /api/warehouse/hazard-items`** 接口。
-   **数据变换**：
    -   用户在搜索框输入的**危废品名称**会作为查询参数传递给 `GET` 接口。
    -   用户在新增弹窗中填写的**危废品名称**，作为请求体传递给 `POST` 接口。
-   **数据消费**：
    -   本页面创建的危废品，将作为**入库单**页面 (`/warehouse/entry`) 中“选择危废品”下拉框的数据源。
    -   也会被**处置台账**页面 (`/warehouse/disposal`) 用于记录危废品的最终去向。
-   **跨页面数据一致性要求**：
    -   如果本页面删除了一种危废品，而该危废品已经被用于某个入库单，那么入库单页面在选择该物品时可能出现错误或空白。这是一个高风险的数据一致性问题，需要业务流程（`BR-HI-003`）或后端逻辑（软删除）来解决。
