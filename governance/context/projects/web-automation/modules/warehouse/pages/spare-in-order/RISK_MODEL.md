## Part 1: RISK_MODEL — 6-Dimension Risk Analysis

### 1.1 Business Risks

| 风险ID | 描述 | 严重度 | 缓解措施 |
|--------|------|--------|----------|
| BR-SIO-01 | 新增入库单未成功提交，业务数据丢失 | P0 | `test_add_in_order_success` 验证行存在 + count增加 |
| BR-SIO-02 | 删除操作误删有效入库单 | P0 | `test_delete_created_in_order` 验证行不再存在 |
| BR-SIO-03 | 审批链缺失/变更导致新增不能生效 | P0 | 新增后检查审批状态（当前无覆盖） |
| BR-SIO-04 | 搜索功能异常，无法检索已有入库单 | P1 | `test_search_by_handler` 覆盖经办人搜索 |
| BR-SIO-05 | 重置功能异常，搜索条件残留 | P2 | `test_reset_search` 验证重置清除条件 |
| BR-SIO-06 | 查看功能异常，详情展示错误 | P2 | `test_view_first_record` 冒烟覆盖 |
| BR-SIO-07 | 表格列结构变更导致自动化脚本失效 | P2 | `test_columns_count` 监控列数变化 |

## Part 2: BUSINESS_SCENARIOS

### 2.1 Business Goal

管理备品备件的入库流程，支持新增入库单（含审批链审批）、查询浏览历史入库记录、查看详情，确保备件入库流程可追溯、可控制。

### 2.2 Roles & Journeys

| 角色 | 操作旅程 | 涉及功能 |
|------|----------|----------|
| 仓库管理员 | 登录 → 导航至入库管理 → 新增入库单 → 填写经办人 → 提交 → 待审批 | 新增，填写 |
| 仓库主管 | 登录 → 导航至入库管理 → 浏览入库单 → 查看详情 | 搜索，查看 |
| 审批人 (admin) | 登录 → 审批中心 → 备件入库审批链 → 会签审批 | 审批（外部流程） |
| 审计人员 | 登录 → 导航至入库管理 → 按日期/经办人搜索 → 查看历史记录 | 搜索，查看 |

### 2.3 Workflows

#### Happy Path: 新增入库单（含审批）
```
1. 导航至入库管理页
2. 点击"新增入库"按钮
3. 对话框弹出，填写经办人
4. 点击"确定"提交
5. 表格新增一行，状态为"待审批"
6. admin登录审批中心，会签通过
7. 状态更新为"已审批"
```

#### Alternative Path: 查看详情
```
1. 导航至入库管理页
2. 点击第一行的"查看"按钮
3. 详情弹窗/抽屉打开，展示完整入库单信息
4. 检查各字段内容
5. 关闭详情
```

#### Alternative Path: 取消新增
```
1. 点击"新增入库"按钮
2. 对话框弹出
3. 点击"取消"按钮
4. 对话框关闭，无新数据行
```

#### Exception Path: 必填校验
```
1. 点击"新增入库"按钮
2. 对话框弹出，不填写经办人
3. 点击"确定"提交
4. 错误信息显示，对话框关闭
5. 无新数据行生成
```

#### Happy Path: 搜索删除
```
1. 使用经办人搜索
2. 表格过滤出对应入库单
3. 点击删除操作
4. 确认后数据行消失
```

### 2.4 Business Rules

| 规则ID | 描述 | 验证点 | 置信度 |
|--------|------|--------|--------|
| BR-SIO-01 | 经办人为必填项 | `test_add_empty_required` error logged | 高 |
| BR-SIO-02 | 新增入库单必须产生新行 | `after >= before + 1` | 高 |
| BR-SIO-03 | 删除入库单必须移除行 | `not is_row_present` | 高 |
| BR-SIO-04 | 取消操作不产生持久化数据 | `not is_row_present` | 高 |
| BR-SIO-05 | 按经办人搜索应精确匹配 | 搜索后验证目标行可见 | 高 |
| BR-SIO-06 | 表格列数在6-12之间 | `test_columns_count` | 高 |
| BR-SIO-07 | 入库单需经审批链审批 | 备件入库审批链配置 | 中（文档说明） |

### 2.5 Data Flow

```
用户输入 → el-input (v-model) → 前端表单对象 → API POST /api/spare-in-order
                                                      ↓
后端校验 → 创建入库单 (状态: 待审批) ← 触发审批流程
                                         ↓
                                admin 会签审批 → 审批通过
                                         ↓
                                状态更新为"已审批"
                                                      ↓
前端刷新 → el-table 重新渲染 → 分页更新
```
