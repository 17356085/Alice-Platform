# BUSINESS_SCENARIOS — personnel / entry-confirm

## 1. 业务目标 (Business Goal)

- **核心业务目标**: 确认持入场申请的承包商人员已实际到达作业现场，完成安保准入流程的最终闭环。
- **次要业务目标**: 为后续生成入场记录、统计现场作业人数提供数据基础。
- **非目标**: 本页面不处理入场申请（由“入场申请”页面处理）、不处理入场计划。

**与RISK_MODEL映射**:
- BS-BUS-001: 保障“确认”操作的唯一性和准确性 (对应 RISK-BUS-001, RISK-BUS-002)。
- BS-BUS-002: 确保只有安保人员能执行此操作 (对应 RISK-AUTH-002)。

## 2. 角色与旅程 (Roles & Journeys)

- **角色列表**:
  - **安保人员 (Security Guard)**: 核心角色。负责现场核验，进行确认操作。
  - **承包商管理员 (Contractor Admin)**: 可查看入场申请状态，但无确认权限。
  - **现场负责人 (Site Manager)**: 可查看入场人数统计，但无确认权限。

- **每个角色的典型业务旅程**:
  - **安保人员**:
    1. 登录系统，导航至“入场确认”页面。
    2. (旅程A) 搜索特定承包商，确认名单后，单行点击“确认入场”。
    3. (旅程B) 批量核对完一个承包商的名单后，勾选多行，点击“批量确认入场”。
    4. 系统提示成功，列表状态更新为已确认。
  - **承包商管理员**:
    1. 登录系统，导航至“入场确认”页面。
    2. 只能查看列表数据，确认按钮置灰或不可见。
    3. (旅程C) 查看自己所属承包商的入场确认进度。

## 3. 业务流程 (Workflows)

- **跨页面主流程（本页面在流程中的位置）**:
  `入场申请 (EntryApply)` → `入场审批 (EntryApprove)` → **`入场确认 (EntryConfirm)`** → `入场记录 (EntryRecord)`

- **Happy Path (HP)**: 安保人员查看列表 → 选择人员 → 点击“确认入场” → 弹窗确认 → 点击“确定” → Toast “确认成功” → 列表状态更新。
- **Alternative Path 1 (AP1) - 批量确认**: 安保人员勾选多行 → 点击“批量确认入场” → 弹窗确认 → 点击“确定” → Toast “批量确认成功” → 列表状态更新。
- **Alternative Path 2 (AP2) - 取消操作**: 安保人员点击“确认入场” → 弹窗 → 点击“取消” → 弹窗关闭，状态无变化。
- **Alternative Path 3 (AP3) - 无数据搜索**: 安保人员输入不存在的关键词 → 点击“搜索” → 表格显示“暂无数据”。
- **异常中断及恢复流程**:
  - **接口超时**: 安保人员点击确认 → Toast “网络异常，请稍后重试” → 列表数据未变更，可重新尝试。
  - **并发冲突**: 后台已确认，安保人员再次点击 → Toast “记录状态已变更，请刷新后重试” → 列表需刷新。

## 4. 业务规则 (Business Rules)

- **状态流转规则**: 记录状态只能从 `待确认` → `已确认`，不可逆。
- **触发规则**: 【确认】成功后，触发后台生成入场记录，并更新人员入场状态。
- **操作权限规则**: 只有 `security_guard` 角色可执行确认操作。`contractor_admin` 和 `site_manager` 角色不可操作。
- **数据可见性规则**: `contractor_admin` 只可查看自己归属承包商的数据。`security_guard` 和 `site_manager` 可查看所有承包商数据。
- **弹窗显示规则**: 弹窗抬头应显示“确认入场”，内容应包含“请确认以下人员入场: [承包商名称] - [人员姓名]”。

## 5. 数据流 (Data Flow)

- **数据来源 (From)**: 入场审批页 (`entry-approve`) 审批通过后产生的 `待确认` 入场申请记录。
- **数据变换 (Transform)**: 本页面将记录的 `状态` 从 `PENDING_ENTRY` 更新为 `CONFIRMED_ENTRY`；生成 `入场确认人`、`确认时间` 字段。
- **数据消费 (To)**:
  - `入场记录` 页面 (`entry-record`): 读取已确认的入场记录用于展示。
  - 报表模块 (`personnel-report`): 读取确认时间用于统计月度/周度入场人次。
- **跨页面数据一致性要求**: 本页面点击确认后，`入场申请` 页面的该条记录状态应同步更新。
