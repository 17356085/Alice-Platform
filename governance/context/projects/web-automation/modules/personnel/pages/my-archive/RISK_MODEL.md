```markdown
# BUSINESS_SCENARIOS: personnel / my-archive (我的档案)

> **版本**: 1.0 | **最后更新**: 2026-06-18 | **维护者**: test-design-agent
> **目标**: 从业务视角描述本页面需要验证的核心场景，作为测试设计的输入。

## 1. 业务目标 (Business Goal)

- **核心业务目标**: 提供一个安全、自助的渠道，让员工能够核对、更新自己的关键个人档案信息（如联系电话、紧急联系人、证件有效期），确保HR系统数据的准确性和及时性。
- **次要业务目标**: 提供历史变更记录的审计追溯能力，支持员工自查档案变动历史。
- **非目标**: 本页面**不**是HR修改员工档案的入口，HR修改员工档案应通过`personnel/employee-management`页面进行。

## 2. 角色与旅程 (Roles & Journeys)

### 2.1 角色列表
| 角色 | 职责边界 | 备注 |
| :--- | :--- | :--- |
| **employee (员工)** | 只能查看和编辑**自己**的档案。不能编辑部门、职位等敏感字段。 | 主要角色 |
| **hr (人力资源)** | 可以查看**自己**的档案，但通常不通过此页面管理他人信息。该页面应限制对敏感字段的编辑。 | 次要角色 |

### 2.2 典型业务旅程

**旅程1: 普通员工更新联系方式**
1. **员工**收到系统通知（如生日邮件）提醒更新个人信息。
2. 员工登录系统，点击导航栏“我的档案”。
3. 页面加载后，默认展示“个人基本信息”Tab。
4. 员工点击“编辑资料”按钮。
5. 弹窗出现，员工将“手机号”从旧号修改为新号，点击“保存”。
6. 系统提示“保存成功”，页面刷新后显示新手机号（中间4位隐藏）。
7. **安全验证**: 3分钟后，员工尝试用新手机号找回密码，收到系统短信验证码。

**旅程2: 员工查看并确认档案变更**
1. 员工怀疑上周某次系统同步导致了档案字段（如“职位”）变更。
2. 员工登录系统，进入“我的档案”。
3. 点击“档案变更记录”Tab。
4. 在筛选区，选择时间范围（如“上周一”到“上周日”），点击“查询”。
5. 表格显示该时间段内所有变更记录，员工查看到“职位”字段的`origin`为“初级工程师”，`new`为“工程师”，操作人为“system”。
6. 员工确认这是一次正常的自动晋升操作。

## 3. 业务流程 (Workflows)

### 3.1 Happy Path: 成功更新个人手机号
1. **前置条件**: 员工已登录，且账号状态正常。
2. **操作流程**:
   - 进入页面 -> 基本信息Tab -> 点击“编辑资料”
   - 编辑弹窗打开，修改“手机号”为合法有效号码
   - 点击“保存”
3. **预期结果**:
   - 前端显示“保存成功”提示
   - 接口返回200，后端数据更新成功
   - 页面刷新，新手机号在前端展示（中间4位隐藏）

### 3.2 Alternative Path 1: 保存时发生网络错误
1. **前置条件**: 同Happy Path。
2. **操作流程**:
   - 在点击“保存”按钮的瞬间，网络断开或服务器500错误。
3. **预期结果**:
   - 前端按钮恢复为非loading状态
   - 弹窗未关闭，显示错误提示“保存失败，请重试”
   - 数据在数据库中应保持一致，不能出现部分字段更新成功的情况。
   - 用户可再次点击“保存”重试。

### 3.3 Alternative Path 2: 尝试修改已被HR限制的字段
1. **前置条件**: 员工角色。
2. **操作流程**:
   - 进入编辑弹窗。
   - 尝试修改“部门”下拉框的值（但动态列表只包含当前部门）。
   - 或通过浏览器调试工具手动修改`<input>`的value并提交。
3. **预期结果**:
   - 前端下拉框无法选择其他部门。
   - 接口收到请求后，返回`403 Forbidden`或`business error`，拒绝更新。
   - 前端弹窗显示“您没有权限修改此字段”。
```
