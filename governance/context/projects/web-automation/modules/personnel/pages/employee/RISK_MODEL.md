### 产出物 2: BUSINESS_SCENARIOS.md

# 业务场景模型: personnel-employee

## 1. 业务目标 (Business Goal)

- **核心业务目标**: 支撑企业员工**全生命周期管理**（入职 → 转正 → 转岗 → 离职）。这是所有HR业务的基石。
- **次要业务目标**:
  - 快速、批量处理员工入、离职（特别是招聘季、裁员潮）。
  - 提供员工基本信息（联系方式、部门、岗位）给下游系统（请假、发薪、考勤）。
- **非目标**:
  - 此页面不处理薪资计算（那是薪资模块的职责）。
  - 此页面不管理员工培训记录（那是培训模块的职责）。

## 2. 角色与旅程 (Roles & Journeys)

| 角色 | 职责边界 | 典型业务旅程 |
|---|---|---|
| **HR管理员** | 负责全公司的员工信息增、删、改、查，及导入导出。 | 1. 批量导入新员工（入职工表）→ 2. 点击“导入”，上传CSV → 3. 查看导入报告（成功/失败原因）→ 4. 手动修改失败行 → 5. 提交生效。 |
| **部门经理** | 只能查看、编辑本部门员工信息（如修改联系电话、紧急联系人）。 | 1. 搜索本部门员工（按姓名） → 2. 点击“编辑” → 3. 修改员工手机号 → 4. 点击“保存”。 |
| **普通员工** | 只能查看自己的个人档案，不能编辑。 | 1. 登录系统 → 2. 在“我的档案”菜单点击进入 → 3. 核对信息。 |
| **系统管理员** | 负责权限配置、数据恢复，不负责业务数据。 | 1. 回收站功能中，原HR误删的员工，进行数据恢复。 |

## 3. 业务流程 (Workflows)

- **跨页面主流程 (Happy Path - 员工入职)**:
    `招聘页面 (审批通过)` → `员工管理 (新增)` → `员工详情 (填写合同、银行卡等)` → `薪资模块 (计算工资)`
- **分支流程 (员工转岗)**:
    `员工管理 (搜索员工)` → `点击“转岗”` → `选择新部门、新岗位` → `系统自动更新组织架构` → `相关审批流触发`
- **异常中断及恢复流程 (员工离职，关联数据核对)**:
    `员工管理 (搜索员工)` → `点击“离职”` → `系统检查是否关联未结清项目/宿舍` → `若有关联，弹出提醒“该员工有未完成项目，是否强制离职？”` → `HR选择“强制离职”，系统自动标记为离职状态` → `项目模块显示“项目负责人已离职”状态`。

## 4. 业务规则 (Business Rules)

| ID | 规则描述 | 类型 | 置信度 |
|---|---|---|---|
| BR-001 | **手机号/邮箱唯一**：新增或编辑时，系统必须校验手机号和邮箱是否已被其他在职员工占用。 | 业务校验规则 | ✅ 已确认 |
| BR-002 | **离职不可逆**：一旦员工状态标记为“离职”，该记录必须保留，不可物理删除（仅可软删除）。 | 状态流转规则 | ✅ 已确认 |
| BR-003 | **默认部门继承**：新增员工时，若不选择部门，默认继承其直属上级的部门。 | 业务校验规则 | ⚠️ 推断 |
| BR-004 | **组织架构继承**：员工转岗后，该员工原部门下的子部门（如有）自动归属权转移到新部门负责人或系统管理员临时管理。 | 触发规则 | ⚠️ 推断 |

## 5. 数据流 (Data Flow)
