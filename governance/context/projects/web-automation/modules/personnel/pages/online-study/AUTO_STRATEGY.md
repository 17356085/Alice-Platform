### 2. PageObject 拆分方案

根据“一个页面一个 Page 类，复杂弹窗独立”的原则，建议如下拆分：

```
OnlineStudyPage (现有)
├── search() / reset_search()
├── get_table_data()
├── get_pagination_info()
└── open_new_course_dialog()
└── open_edit_course_dialog(course_name)

OnlineCourseDialog (新建)  <-- 从 OnlineStudyPage 提取
├── fill_form(course_data: dict)
├── get_form_data() -> dict
├── save()
├── cancel()
└── is_visible() -> bool

DeleteConfirmDialog (新建)  <-- 复用通用确认框组件
├── confirm()
└── cancel()
```

**拆分原因**:
- `OnlineCourseDialog` 封装了新建/编辑课程弹窗的所有操作（含表单、上传、开关等），业务复杂度高，独立成类便于重用和维护。
- `DeleteConfirmDialog` 是一个通用的确认弹窗，可能在多个页面出现，独立成公共组件。

**PageObject 职责边界**:
- `OnlineStudyPage`: 负责页面导航、搜索区、表格区、分页区的操作。
- `OnlineCourseDialog`: 负责弹窗的显示、表单填充、提交、取消等操作。
- `DeleteConfirmDialog`: 负责确认/取消操作。

### 3. 公共组件复用分析

| 操作 | 所属区域 | 能否复用 BasePage/已有方法 | 说明 |
|:-----|:---------|:------------------------|:-----|
| 页面导航 | 整个页面 | ✅ `navigate_to()` | 菜单导航，直接继承 |
| 输入框填充 | 搜索区/表单 | ✅ `fill_input()` | 通用 |
| 点击元素 | 全局 | ✅ `click_element()` | 通用 |
| 选择下拉 | 搜索区/表单 | ✅ `click_by_text()` | 需确保 `el-select` 的 Dropdown 已展开 |
| 点击日期范围 | 搜索区 | ⚠️ 无通用方法 | 需要先 `clear()`，再 `fill()`，可能需要额外处理日期面板 |
| 上传文件 | 表单 | ⚠️ `upload_file()` | `el-upload` 使用隐藏的 `<input type="file">`，需特殊定位 |
| 判断元素存在/可见 | 全局 | ✅ `is_element_visible()` | 通用 |
| 等待 loading 消失 | 全局 | ✅ `wait_loading_disappear()` | 通用 |

**是否需要扩展 ElementPlusHelper**:
否。当前操作均能通过 `BasePage` 和 `WebUILibrary` 覆盖，无需额外封装。

### 4. 等待策略建议

该页面特有的异步行为及应对策略：

| 异步场景 | 触发条件 | 建议等待方式 | 说明 |
|:---------|:---------|:-------------|:-----|
| **表格加载** | 搜索、重置、翻页、新建/编辑/删除后 | `wait_loading_disappear(TABLE_LOADING)` | 必须等待 loading 消失再操作 |
| **弹窗打开** | 点击“新建课程”或“编辑” | `wait_element_visible(DIALOG_COURSE)` | 建议额外等待弹窗动画完成 (500ms) |
| **弹窗关闭** | 点击“取消”或“确定” | `wait_element_invisible(DIALOG_COURSE)` | 确保弹窗完全关闭后再操作表格 |
| **下拉菜单展开** | 点击 `el-select` | `wait_element_visible(DROPDOWN)` | `el-select` 的 Dropdown 是异步渲染到 body 的 |
| **分页数据刷新** | 切换每页条数或页码 | `wait_loading_disappear(TABLE_LOADING)` + 验证数据更新 | 确保新数据已加载 |
| **上传文件后** | 选择文件 | `wait_attribute(CSS_SELECTOR, "class", "is-ready")` 或等待上传接口返回 | 上传状态变化异步，需针对性等待 |

**封装的等待方法建议**:
可以考虑在 `OnlineStudyPage` 中添加一个通用等待方法 `wait_for_table_stable()`，内部组合 `wait_loading_disappear(TABLE_LOADING)` 和 `implicitly_wait(1)` (等待 Vue 响应式更新)，用于所有表格操作后。

**预估成本**:
- 开发时间 (T开发): **15 小时** (<- 包括编写/调试自动化脚本、提取 Dialog 类、集成 CI)
- 维护成本 (T维护/月): **2 小时** (<- 定位器微调、新增用例、脚本维护)

**手工执行成本**:
- 手工执行一次 (T手工): **15 分钟**
- 每月手工执行次数 (N): **30 次** (假设每天执行一次回归)
- 月度手工执行时间 (T手工-月): 15 分钟 * 30 = **7.5 小时**

**自动化收益计算**:
- 月度自动化执行时间 (T自动-月): **1 小时** (脚本执行 + 失败分析)
- 每月节省时间: (7.5 - 1) - 2 = **4.5 小时**
- **ROI 回本周期**: 15 / 4.5 ≈ **3.3 个月**

**结论**: 自动化方案在 **3-4 个月**内可收回成本，后续每月净省约 4.5 小时。

### 6. 风险及建议

**高风险区域**:
1.  **日期选择器 (`el-date-picker`)**: 操作复杂，涉及 `clear()` 和 `fill()` 的组合，可能导致失败。**建议**: 在 `search()` 方法中对日期处理进行充分的异常捕获和重试。

2.  **文件上传 (`el-upload`)**: 多用于P2用例，且浏览器环境和CI环境差异大。**建议**: 将文件上传作为独立功能测试通过API完成，UI层仅验证状态变化。

3.  **表格操作列按钮定位**: 行索引变化或点击到隐藏的行，可能导致点击失败。**建议**: 在`get_table_data()`后，优先使用 `TextMatch` 定位行，而非行索引。

**一次性操作**:
- 页面迁移/重构后的初始数据导入。
- 相关配置项的首次配置。
以上操作均不需要自动化。
