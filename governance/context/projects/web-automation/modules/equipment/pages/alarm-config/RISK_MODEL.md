### **BUSINESS_SCENARIOS.md**

#### 1. 业务目标
- **核心目标**：运维人员通过配置报警阈值，确保设备运行参数异常时能及时收到通知，避免安全事故。
- **次要目标**：支持不同设备类型（温度、压力、液位）独立配置；支持告警去抖、通知方式等高级功能。
- **非目标**：本页面不处理报警历史查询（归属告警中心模块）、不管理通知接收人。

#### 2. 角色与旅程
| 角色 | 职责边界 | 典型旅程 |
|------|----------|----------|
| 系统管理员（admin） | 配置全局告警策略，可修改所有设备的告警 | 登录→导航至“设备管理→告警配置”→选择设备→修改阈值→保存→验证新阈值生效（可借助模拟数据） |
| 设备运维员（operator） | 仅查看和修改自己负责区域的设备告警 | 登录→默认进入个人看板→点击“查看告警配置”→可能无修改权限（视权限设置）→提交修改需审批 |
| 审核员（auditor） | 仅查看已创建的告警配置，不能修改 | 登录→查看配置日志→核对变更记录 |

#### 3. 业务流程
- **Happy Path**：设备投运（某设备温度首次达到 50°C）→ 运维人员进入告警配置页，设定高温告警阈值60°C → 保存成功 → 设备温度升至65°C → 系统触发告警通知。
- **Alternative Path 1**：设置的去抖时间过长（比如 600秒），温度瞬超65°C但< 600s恢复 → 未产生告警 → 运维人员发现后缩短去抖时间。
- **Alternative Path 2**：设备已拆除（deleted）但告警配置未清除 → 后续新设备 id 复用导致误触发旧告警 → 触发级联删除流程。
- **异常流程**：保存时后端服务中断 → 前端提示“保存失败”，允许用户手动重试；数据不丢失（本地暂存或自动草稿）。

#### 4. 业务规则
- 温度阈值：必须 ≥ 设备物理最低耐温（-40°C）且 ≤ 设备物理最高耐温（200°C）。*可验证：输入300°C应被拒绝*。
- 去抖时间：整数，范围 1~3600 秒。*可验证：输入0秒应提示错误*。
- 一个设备最多可配置 5 个告警规则（不同参数或方向）。
- 告警启用后，修改阈值需记录变更日志（保存前、后值、操作人、时间）。
- 设备状态为“离线”时，告警配置禁止修改（需先恢复设备在线）。

#### 5. 数据流
- **数据来源**：设备列表从 `equipment/list` 接口获取（缓存至前端 store）；告警配置从 `alarm/config/{deviceId}` 读取。
- **数据变换**：用户输入经前端校验（数值范围、非空）→ 组织成 JSON 提交到 `POST /api/alarm/config`；后端记录版本号，保存至数据库 `alarm_config` 表。
- **数据消费**：告警引擎（alert-engine）实时订阅 `alarm_config` 变更，更新内存规则；运维后台读取展示；审计模块读取变更日志。
- **跨页面一致**：在「设备详情页」中看到告警配置摘要，缓存 30s 后刷新；若在告警配置页修改变更，需通知设备详情页刷新（事件总线或 stomp）。
