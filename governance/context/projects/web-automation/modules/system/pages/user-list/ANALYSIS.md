## 1. 需求理解

### 1.1 一句话概括

> **为系统管理员提供平台账号的全生命周期管理能力——创建、查看、编辑、角色分配、密码重置、启停用及删除，支撑 RBAC 权限体系的用户入口。**

### 1.2 需求来源追溯

| 来源 | 类型 | 内容 |
|------|------|------|
| `test_user_management.py` | 自动化用例（15 条） | 页面展示、分页、CRUD、搜索、校验、批量操作 |
| `UserManagePage.py` | Page Object（~1,250 行） | 完整 UI 元素定位 + 操作方法 |
| `PROJECT_CONTEXT.md §4` | 菜单树 | 用户管理归属「系统管理」一级菜单 |
| `PROJECT_CONTEXT.md §5.1` | RBAC 权限模型 | 用户→角色→菜单/按钮权限 的五层体系 |
| `PROJECT_CONTEXT.md §5.2` | 权限标识 | `system:user:add / edit / remove / list / resetPwd` |
| `PROJECT_KNOWLEDGE.md §2.1` | 功能描述 | "增删改查、重置密码、批量删除、角色分配" |
| `PROJECT_KNOWLEDGE.md §4.1` | 已测模块 | 用户管理 15 条（Selenium） |

### 1.3 功能全景图（基于代码反推）

```
用户管理模块
│
├── 📋 用户列表（主页面）
│   ├── 搜索筛选栏
│   │   ├── 关键字搜索（用户名/姓名/手机号 — 单框模糊匹配）
│   │   ├── 角色下拉筛选
│   │   ├── 状态单选筛选（全部/启用/禁用）
│   │   ├── [查询] 按钮
│   │   └── [重置] 按钮 → 清空所有筛选条件
│   ├── 工具栏
│   │   ├── [新增] 按钮
│   │   ├── [批量删除] 按钮 (danger)
│   │   └── [导出] / [导入] 按钮（PO 中有定位，未单独测试）
│   ├── 数据表格
│   │   ├── 列：☐ | 用户名 | 姓名 | 手机号 | 部门 | 角色 | 状态 | 创建时间 | 操作
│   │   └── 行操作按钮：[编辑] [分配角色] [更多▾]
│   │       └── 更多▾：[删除] [重置密码]
│   └── 分页栏
│       └── 共 N 条 | 上一页/下一页
│
├── ➕ 新增用户（Dialog 弹窗）
│   └── 字段：用户名* / 姓名* / 密码* / 确认密码* / 手机号 / 部门* / 角色 / 岗位 / 邮箱
│
├── ✏️ 编辑用户（Dialog 弹窗，复用新增弹窗）

│
├── 🔑 分配角色（Dialog 弹窗 — Checkbox Group）
│   └── 列出系统全部角色，勾选分配
│
├── 🔄 重置密码（MessageBox 确认）
│   └── 确认后生成新默认密码，Toast 展示
│
└── 🗑️ 删除（MessageBox 确认）
    ├── 单条删除（行内「更多▾」→「删除」）
    └── 批量删除（勾选多行 → 工具栏「批量删除」）
```

## 4. 风险识别

### 4.1 需求模糊点

| 编号 | 模糊点 | 影响 | 建议 |
|:---:|------|:---:|------|
| **AMB-001** | **密码最小长度/复杂度要求不明确**：测试中用 `123456`，但生产环境可能有更严格的策略 | 新增/重置密码的校验规则可能不匹配实际策略 | 确认后端密码策略配置（参数设置中的密码规则），补充边界测试 |
| **AMB-002** | **用户名允许的字符范围不明确**：2-20 字符但未说明是否支持中文/特殊字符 | 用中文或特殊字符作用户名可能导致登录异常 | 确认用户名规则（字母+数字+下划线？），补充字符集边界测试 |
| **AMB-003** | **"更多▾"菜单项的显隐逻辑不明确**：是否所有用户行都显示"删除"和"重置密码"？admin 用户能否被删除？ | admin 账号被误删将导致系统不可用 | 确认自我保护机制，补充分级用户的操作权限矩阵测试 |
| **AMB-004** | **批量删除是否有数量上限**：一次性删除 100+ 用户的行为未定义 | 大批量删除可能导致超时或部分成功 | 确认后端批量删除实现（事务/分批），设定合理的测试上限 |
| **AMB-005** | **角色分配时是否可以清空所有角色**：即用户无任何角色是否允许 | 无角色用户可能无法使用任何功能（白屏或无限跳转） | 确认是否允许零角色用户存在，补充边界测试 |

### 4.2 逻辑矛盾/遗漏

| 编号 | 问题 | 严重程度 | 建议 |
|:---:|------|:---:|------|
| **CON-001** | **手机号校验仅为格式校验（长度=11），不验证号段合法性**：`12345678901` 这种明显非法的号码可通过校验 | 🟡 中 | 确认是否需要运营商号段正则校验（如 `^1[3-9]\d{9}$`） |
| **CON-002** | **密码字段在编辑弹窗中同时存在"密码"和"确认密码"**：但测试中仅填入密码，未显式处理确认密码。若两密码不一致时的行为未测试 | 🟡 中 | 补充确认密码一致性校验的测试用例 |
| **CON-003** | **新增成功的 Toast 断言条件过宽**：`"成功" in toast or "新增" in toast or toast` — 只要 Toast 非空即算通过，可能漏掉错误提示 | 🟢 低 | 收紧断言，如必须包含"成功"或"新增成功" |
| **CON-004** | **搜索框 placeholder 包含"搜索用户名/姓名/手机号"**：暗示支持多字段搜索，但未测试搜索"手机号的一部分"（如中间4位）的结果 | 🟢 低 | 补充搜索精度测试（部分匹配 vs 完全匹配） |

### 4.3 与技术架构的冲突

| 编号 | 问题 | 严重程度 | 建议 |
|:---:|------|:---:|------|
| **TECH-001** | **Vue 异步渲染导致弹窗元素定位偶发失败**：PO 中大量使用 `WebDriverWait` + JavaScript 点击的降级策略，说明页面渲染不稳定 | 🟡 中 | 记录不稳定用例，增加重试机制或等待策略调优 |
| **TECH-002** | **Element Plus Teleport 导致弹窗 DOM 挂载在 body 下**：定位器需处理多种弹层容器（`el-overlay`/`el-dialog__wrapper`/`el-popper`），维护成本高 | 🟡 中 | 封装弹窗定位工具方法，统一弹窗操作入口 |
| **TECH-003** | **部门 TreeSelect 的父子节点可选性不一致**：PO 中有 `select_dialog_first_valid_option` 方法专门处理"父节点不可选"的场景，说明数据结构上存在限制 | 🟢 低 | 记录部门树中哪些节点可选/不可选，覆盖边界场景 |
| **TECH-004** | **双 Token 刷新期间用户管理操作的行为未测试**：若 accessToken 在表单提交期间过期，静默刷新是否中断用户操作 | 🟡 中 | 补充 Token 过期边界场景的测试（可在配置中缩短 Token 有效期模拟） |
| **TECH-005** | **测试数据清理依赖代码执行顺序**：TC-015 批量删除承担清理职责，若在此之前用例失败或跳过，清理不执行 | 🟡 中 | conftest 已有 `_teardown_user_module` 兜底，但需验证其在所有失败模式下的可靠性 |

## 5. 测试策略建议

### 5.1 策略选择

| 维度 | 评估 | 结论 |
|------|------|------|
| 接口层（API） | Web 端 pytest 已有覆盖，小程序无需重复 | 🟢 **沿用现有** |
| UI 层（Selenium） | ✅ 15 条用例已覆盖核心 CRUD + 搜索 + 分页 | 🟢 **增强现有** |
| 手工测试 | 权限矩阵、异常场景、跨模块联动 | 🟡 **补充** |

> **推荐策略**: **混合策略 — 自动化为主（80%）+ 手工补充（20%）**

### 5.2 策略详解

```

│                    测试金字塔 — 用户管理                    │

│        │  ~5 条   │     跨模块联动、探索性测试             │
│        └────┬────┘                                      │
│             │                                           │
│    ┌────────┴────────┐                                  │
│    │   UI 自动化      │  ← Selenium 15 条(已有)           │
│    │   +8 条待补充     │     + 角色分配/启停用/导入导出     │
│    └───────┬─────────┘                                  │
│            │                                            │
│  ┌─────────┴──────────┐                                 │
│  │   API 接口测试       │  ← pytest 通过 Web 端间         │
│  │   (Web端已覆盖)      │     接覆盖用户管理 API           │
│  └────────────────────┘                                 │
└─────────────────────────────────────────────────────────┘
```

### 5.3 为什么不是全自动化

| 不适合自动化的场景 | 原因 |
|------|------|
| **数据范围验证** | 需要切换不同 DataScope 角色，验证不同角色看到的用户列表差异 — 需要多账号+复杂数据准备 |
| **跨模块联动** | 删除用户 → 检查审批链是否断裂 → 检查操作日志记录，链路过长，维护成本高 |
| **admin 自我保护** | 验证 admin 用户不可被删除/停用 — 边界条件少，手工一次即可 |

### 7.3 技术实现提示

# 推荐复用现有 Page Object，新增用例只需添加测试方法

# 示例：确认密码不一致校验
def test_016_confirm_password_mismatch(self, driver_setup):
    """SY-USER-016: 新增用户-确认密码不一致校验"""
    user_manage = UserManagePage(driver_setup)
    user_manage.click_add_user_button()
    user_manage.input_dialog_input("用户名", f"test_{datetime.now().strftime('%Y%m%d%H%M%S')}")
    user_manage.input_dialog_input("姓名", "密码测试")
    user_manage.input_password_in_dialog("123456")  # 密码: 123456
    user_manage.input_dialog_input("确认密码", "654321")  # 确认密码: 不同
    user_manage.select_dialog_first_valid_option("部门")
    user_manage.click_dialog_confirm()
    error = user_manage.get_form_error_text()
    assert "密码" in error and ("不一致" in error or "不相同" in error)

# 示例：角色分配流程
def test_019_assign_role(self, driver_setup):
    """SY-USER-019: 分配角色-勾选新角色并提交"""
    # 前提：已通过闭环数据创建 test 用户
    user_manage = UserManagePage(driver_setup)

    user_manage.click_assign_role_dialog_confirm()
    toast = user_manage.get_toast_text()
    assert "成功" in toast or "修改" in toast
