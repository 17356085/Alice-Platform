### 自动化优先级建议

| 测试点/用例 | 自动化优先级 | 自动化方式 | 备注 |
|-------------|--------------|------------|------|
| 页面加载与显示(正常/空数据/权限受限) | **高** | UI自动化 | 基础功能，回归测试必需。 |
| 权限验证(操作员无配置权限) | **高** | API + UI自动化 | 关键安全红线，API验证成本低，UI验证覆盖前端展示。 |
| 业务场景-报警响应流程(E2E) | **高** | API + UI混合 | 核心业务流程，需要自动化验证。 |
| 业务场景-误操作与撤销(E2E) | **低** | 手动测试 | 依赖用户交互确认(弹窗、双击)，自动化脚本复杂且脆弱，收益不高。 |
| 异常-数据延迟 | **中** | Mock + UI自动化 | 需要Mock后端数据推送延迟，自动化脚本较复杂。 |
| 异常-Token过期 | **高** | API自动化 | 核心安全场景，API自动化成本低，执行快。 |

### 不建议自动化的场景

1.  **TD-MON-BIZ-003 / TC-MON-005 (误操作与撤销)**: 该场景涉及复杂的用户意图（双击、阅读确认框内容、点击取消），自动化脚本需要模拟光标移动和双击动作，且对UI布局变化敏感。手动测试可以更真实地覆盖用户误操作的心理和行为特征，且执行成本远低于维护这样一个复杂且脆弱的自动化脚本。
2.  **TD-MON-SRC-002 (搜索不存在罐)**: 搜索功能通常已经很稳定且自动化成本低，但该特定场景（搜索不存在）的价值较低，手动测试即可覆盖。
3.  **TD-MON-TBL-001 (报警列表分页)**: 如果UI组件稳定且数据量固定，可以自动化。但若分页组件复杂或数据量动态变化，手动测试可以更快地检查边界情况（如首次加载、中间页、最后几页的显示逻辑），自动化脚本容易因数据问题失败。手动测试即可。
