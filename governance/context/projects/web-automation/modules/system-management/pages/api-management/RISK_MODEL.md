### 1. 业务风险 (Business Risk)

| 风险ID | 风险描述 | 风险等级 | 缓解措施 | 关联自动化覆盖 |
|---|---|---|---|---|
| **RISK-BIZ-001** | **误删API导致角色权限失效**: 删除了被角色引用的API，该角色的所有用户将失去对应功能的访问权限，属于严重业务事故。 | **P0** | 1. 删除前必须二次确认并明确提示“该API已被X个角色引用”。<br>2. 后端应实现软删除或引用检查，禁止删除被引用的API。<br>3. 审计日志记录删除操作详情。 | ⚠️ 现有自动化仅覆盖单条删除的Happy Path（选择一条数据→点击删除→确认），未覆盖“被引用API删除”的阻断场景。 |
| **RISK-BIZ-002** | **批量操作(如批量启用/禁用)导致大面积业务中断**: 由于粗心或配置错误，管理员批量禁用了核心业务API，导致大面积服务不可用。 | **P1** | 1. 批量操作应增加二次确认弹窗，并列举受影响API数量。<br>2. 提供“已启用/禁用”筛选，方便管理员精确选择。 | ❌ 未覆盖批量操作场景。 |

### 2. 权限风险 (Permission Risk)

### 业务场景详解 (补充说明)

#### 1. 业务目标
*   **核心目标**: 提供集中式API定义和配置管理，作为系统权限控制的基础数据源。
*   **次要目标**: 允许管理员灵活启用/禁用API，进行紧急维护和灰度发布。
*   **非目标**: 本页面不解决API的版本管理、调用日志分析、性能监控等问题。

#### 2. 角色与旅程
*   **系统管理员 (核心用户)**:
    *   **典型旅程**: 登录系统 → 进入“系统管理” → 点击“API管理” → 在列表页查询/查看/新增/编辑/删除/启用禁用API → 不通过其他页面直接完成所有操作。
*   **普通运维人员 (受限用户)**:
    *   **典型旅程**: 无直接访问API管理页面的权限，仅能通过其他权限分配好的功能间接使用API。

#### 3. 业务流程
*   **主流程 (Happy Path)**:
    1.  管理员登录系统。
    2.  点击“系统管理” → “API管理”。
    3.  页面正常展示API列表 (搜索区、分页、表格)。
    4.  点击“新增”按钮，填写表单，点击“确定”。
    5.  弹出成功提示，列表刷新，新增API出现在第一页。
*   **Alternative Path 1 (编辑并启用/禁用)**:
    1.  在列表中点击某条API的“编辑”按钮。
    2.  修改信息，切换状态开关，点击“确定”。
    3.  列表刷新，该API状态更新。
*   **Alternative Path 2 (删除并产生级联影响)**:
    1.  在列表中选中一条被角色引用的API，点击“删除”。
    2.  弹窗提示“该API已被X个角色引用，删除后对应角色的用户将无法访问该API，确定删除吗？”
    3.  管理员点击“确定”。
    4.  列表刷新，该API消失。
    5.  相关角色的用户登录后，发现无法访问原API对应的功能。
*   **异常流程**:
    *   **Token过期**: 用户在页面上停留过久，Token过期。点击任意操作按钮（编辑、删除等），接口返回401，前端自动跳转到登录页。
    *   **网络异常**: 用户点击保存时网络断开，页面一直显示“保存中...”的loading状态，超时后提示“网络异常，请稍后重试”。

#### 4. 业务规则
*   **规则1: 唯一性约束**: 同一系统中，`API路径 + 请求方式` 组合必须唯一。
*   **规则2: 路径格式**: API路径必须以 `/` 开头，且不能包含空格或特殊字符（如 `#`, `?`）。
*   **规则3: 引用保护**: 已被角色引用的API，不能直接物理删除，需给出提示并由管理员二次确认。
*   **规则4: 状态继承**: 当API被禁用时，通过该API鉴权的页面/功能应立即不可用，无需重启服务。
*   **规则5: 权限点依赖**: `编辑`、`删除`、`启用/禁用` 这三个操作依赖 `system:api:edit` 权限点，`新增` 依赖 `system:api:create`。

#### 5. 数据流
*   **创建**: 前端 `dialog-apiName`, `dialog-apiPath`, `dialog-requestMethod`, `dialog-status` → `POST /system/api` → 后端校验 → 写入 `system_api` 表。
*   **读取**: `GET /system/api?page=1&size=20` → 读取 `system_api` 表 → 返回列表数据。
*   **消费**:
    *   **角色管理**: 读取 `system_api` 表，构建权限树。
    *   **用户鉴权**: 用户登录后，后端根据 `user_role` → `role_api` → `system_api` 链路，生成权限标识列表，前端用于控制按钮/页面可见性。

#### 6. 业务风险→场景映射
*   **RISK-BIZ-001 (误删)** → **BS-BIZ-002** (已映射)
*   **RISK-PERM-001 (越权)** → **BS-RL-002** (已映射)，**BS-RULE-002** (已映射)
*   **RISK-DATA-001 (重复)** → **BS-FLOW-001**, **BS-FLOW-002** (已映射)
*   **缺失覆盖**: `RISK-INT-001` (Token过期) 和 `RISK-INT-002` (接口超时) 对应的业务场景（如Token过期后重新登录），在现有业务场景表中未显式列出，待补充到业务流程的“异常流程”中。
