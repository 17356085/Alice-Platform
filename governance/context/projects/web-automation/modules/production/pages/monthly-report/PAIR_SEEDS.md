# PAIR_SEEDS — 生产月报表

> 结对同学：AI Pilot Test | 日期：2026-06-22 | 需求ID：TAPD-PROD-REPORT-001

## 种子场景

### SEED-001：跨月导航后生成报表数据一致性

- **优先级**: P0
- **类型**: positive
- **业务背景**: 月份导航使用自定义箭头组件（非 el-date-picker），AI 容易低估其测试复杂度。月份切换后，统计卡片和表格必须全部刷新为新月份的数据。如果数据残留上个月的缓存，会导致生产决策错误。
- **前置条件**: 1. 系统中有 2026年5月和6月的数据 2. 默认显示当前月（6月）
- **操作步骤（概要）**:
  1. 打开生产月报表页面
  2. 记录当前月份的 LNG 月产量、液氮月产量数值
  3. 点击"←"切换到上一月（5月）
  4. 等待页面刷新
  5. 对比两月数据
- **预期结果**: 1. 月份显示从"2026年6月"变为"2026年5月" 2. 统计卡片数值全部更新（与6月不同） 3. 表格数据对应5月 4. 再切回6月时数据恢复
- **关键检查点**: 月份导航是自定义组件，DOM 更新时机可能滞后。需验证 `current-month` 的 textContent 与统计卡片的数值同步变化。

### SEED-002：生成报表按钮的启用/禁用逻辑

- **优先级**: P0
- **类型**: boundary
- **业务背景**: "生成报表"按钮初始为 disabled 状态，切换月份后才启用。AI 容易忽略按钮状态转换逻辑——如果按钮一直 disabled，整个报表功能不可用。这种状态机逻辑是前端 Bug 高发区。
- **前置条件**: 1. 页面首次加载 2. 默认月份已有数据
- **操作步骤（概要）**:
  1. 页面加载后检查"生成报表"按钮 → 预期为 disabled
  2. 点击"→"切换到下一月
  3. 检查按钮状态 → 预期变为 enabled
  4. 点击"生成报表"
  5. 等待报表生成
  6. 再次切换月份 → 按钮变回 disabled
  7. 再切回 → 按钮 enabled
- **预期结果**: 1. 初始 disabled 2. 切换月份后 enabled 3. 生成后保持 enabled（当前月已生成） 4. 切换到未生成的月份 → disabled
- **关键检查点**: 状态转换是否在所有路径下正确。连续快速切换时是否存在竞态（disabled → enabled → disabled）。

### SEED-003：导出和打印功能的数据完整性

- **优先级**: P1
- **类型**: positive
- **业务背景**: 导出和打印是月报的核心使用场景。AI 通常只测"点击导出按钮，文件下载成功"，但不会验证导出内容是否与页面展示一致。生产中曾出现"导出 Excel 缺少最后一列"的 Bug。
- **前置条件**: 1. 已生成某月报表 2. 页面展示完整（统计卡片 + 四个分类表格）
- **操作步骤（概要）**:
  1. 点击"导出"按钮
  2. 等待文件下载完成
  3. 打开下载的 Excel 文件
  4. 对比 Excel 中的 LNG 月产量数值与页面 stat-lng-monthly 的数值
  5. 检查 Excel 是否包含全部四个分类（产品/原料/公辅工程/冷剂消耗）
- **预期结果**: 1. Excel 数值与页面展示完全一致 2. 四个分类表格均包含 3. 列头完整 4. 文件名包含月份标识
- **关键检查点**: 导出格式（xlsx/csv？），文件名规范，大数据量时是否超时。

## 补充说明

- **特殊业务规则**: 月份导航器是自定义组件（`month-nav` / `current-month`），不是 el-date-picker。定位需用 class selector（`.month-nav .btn-prev` / `.current-month`）。月份切换后需等待统计卡片数据刷新（`WebDriverWait` 检测 `stat-lng-monthly` 数值变化）。
- **已知坑点**: 报表生成是异步操作——点击"生成报表"后可能需要数秒，期间按钮应显示 loading 状态。如果用户快速双击，可能触发重复生成的 Bug。
- **不需要测试的场景**: 报表打印的纸张布局（Selenium 无法验证打印预览），报表数据的业务准确性（需要业务人员核对）。
