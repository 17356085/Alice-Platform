# 业务场景模型: approval-history

## 1. 业务目标
- **核心目标**: 用户能够快速准确地查找到自己参与的历史审批记录，并执行合法的后续操作（如查看详情、撤回未完成的申请）。
- **次要目标**: 管理员能够监控所有审批历史，用于审计和运维。
- **非目标**: 本页面不负责发起新审批或修改审批流程。

## 2. 角色与旅程
| 角色 | 职责边界 | 典型旅程 |
|------|----------|----------|
| 普通用户（申请人） | 只能查看自己提交的审批历史 | 登录→导航到审批历史→筛选时间→查看详情→若状态为待审批则点击撤回→确认撤回→查看历史状态变为已撤回 |
| 审批人 | 只能查看自己审批过的历史 | 登录→导航到审批历史→默认显示待办历史→筛选已完成→点击详情→评论或重新打开 |
| 管理员 | 可查看所有审批历史 | 登录→导航到审批历史→切换为“全部”视图→输入申请人搜索→批量导出历史数据 |

## 3. 业务流程
### Happy Path (普通用户撤回)
1. 用户登录系统
2. 点击“审批历史”菜单 enter approval-history page
3. 页面默认展示最近7天用户提交的审批记录
4. 用户点击状态为“待审批”的记录行
5. 页面弹出详情抽屉，显示撤回按钮
6. 用户点击撤回 → 弹窗确认 → 确认
7. 接口调用成功 → 列表刷新，该记录状态变为“已撤回”

### Alternative Path 1 (筛选+无结果)
1-3 同上
4. 用户设置筛选条件：状态=“已通过”，时间范围=“2022-01-01 ~ 2022-01-02”
5. 接口返回空数据列表
6. 页面显示“暂无审批历史”，建议调整筛选条件

### Alternative Path 2 (管理员批量操作)
1. 管理员登录
2. 进入审批历史，切换到“全部”视图
3. 勾选多条记录（部分可撤回，部分不可）
4. 点击“批量撤回”
5. 接口返回部分成功列表（可撤回的记录撤回成功，不可撤回的提示原因）
6. 页面展示“成功撤回3条，失败2条（原因：流程已结束）”

### 异常中断及恢复
- **网络中断**: 撤回请求发送后网络断开 → 用户收到“请求失败，请检查网络并重试”，刷新后列表状态未变，用户可重新撤回
- **并发冲突**: 两个审批人对同一申请同时操作 → 后一个操作报“数据已被修改，请刷新后重试”

## 4. 业务规则
| 规则ID | 规则描述 | 置信度 |
|--------|----------|--------|
| RULE-001 | 只有“待审批”状态的记录才能操作撤回 | ✅ 已确认 |
| RULE-002 | 撤回操作仅由申请人本人执行，审批人和管理员不可代劳 | ✅ 已确认 |
| RULE-003 | 审批历史的最长保留时间为5年，超时自动归档（不可查看） | ⚠️ 推断 |
| RULE-004 | 历史列表默认按最后更新时间降序排列 | ✅ 已确认 |
| RULE-005 | 管理员可以查看所有申请人的历史，但不可修改 | ✅ 已确认 |

## 5. 数据流
- **数据来源**: 审批历史数据由 submission 页面创建，存储于 `wf_approval_history` 表。
- **数据变换**: 本页面提供查询和查看详情，撤回操作时调用 `/api/workflow/revoke`，后端将对应记录状态置为“已撤回”，并级联更新关联的步骤状态。
- **数据消费**: 审批历史详情中的数据被 dashboard 的“我的审批”统计和报表模块使用。
- **跨页面一致性**: 撤回后，`我的申请` 页面的记录也应同步更新为已撤回。
