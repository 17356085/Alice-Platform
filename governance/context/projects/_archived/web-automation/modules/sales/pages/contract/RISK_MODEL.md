### 产出物 2: BUSINESS_SCENARIOS.md

# 业务场景模型 - 销售合同管理页

## 1. 业务目标 (Business Goal)

- **核心业务目标**：销售员在此页面创建、修改、提交审批、查看销售合同，从报价到最终签署的全过程管理。
- **次要业务目标**：财务/法务人员在此页面审核合同条款，签署人或管理员在此页面进行签署。
- **非目标**：本页面不处理发票开具（由财务模块处理），也不关联客户主数据变更（由CRM模块处理）。

## 2. 角色与旅程 (Roles & Journeys)

| 角色 | 职责边界 | 典型旅程 |
|---|---|---|
| **销售员** | 创建、编辑、提交自己的合同草稿。 | 1. 登录 → 2. 点击“新建合同” → 3. 填写客户信息、合同条款、金额 → 4. 保存草稿 → 5. 提交审批 → 6. 收到审批通过通知 → 7. 查看已签署合同 |
| **销售主管** | 审批自己团队的合同（金额在一定范围内）。 | 1. 登录 → 2. 点击“待审批” → 3. 查看合同详情 → 4. 批准/驳回 → 5. 提交法务审批 |
| **法务/财务** | 审核合同关键条款和法律风险。 | 1. 登录 → 2. 点击“待审批” → 3. 查看合同条款 → 4. 提出修改意见 → 5. 审批通过 |
| **管理员/超级签署人** | 最终签署合同，使其生效。 | 1. 登录 → 2. 点击“待签署” → 3. 确认无误 → 4. 电子签署 → 5. 合同生效 |

## 3. 业务流程 (Workflows)

- **跨页面主流程**：
  - **草稿 → 待审批 → 已批准 → 待签署 → 已签署** (Happy Path)
  - 本页（contract）作为流程的核心，连接报价（quotation）与订单（order）。
- **分支流程（Alternative Path）**：
  1. **驳回流程**：待审批 → 审批人驳回 → 销售员修改 → 重新提交审批。
  2. **作废流程**：已审批但未签署 → 发起人取消 → 合同状态变为“已作废”。
- **异常中断及恢复流程**：
  - 网络中断，编辑中内容自动保存为草稿，下次登录可恢复。
- **流程状态机抽象描述**:
  ```
  [草稿] -> submit -> [待审批] -> approve -> [已批准] -> sign -> [已签署] -> archive -> [已归档]
  [待审批] -> reject -> [驳回] -> re-submit -> [待审批]
  [已批准] -> cancel -> [已作废]
  ```

## 4. 业务规则 (Business Rules)

| 规则ID | 规则描述 | 所属维度 | 影响场景 | 置信度 |
|---|---|---|---|---|
| BR-001 | 合同总金额超过 100 万元人民币，必须经过法务审批。 | 金额阈值规则 | 待审批流程分流 | ⚠️ 推断 |
| BR-002 | 合同状态为“已审批”后，任何修改都必须生成新版本并重新审批。 | 版本与审批规则 | 修改/版本控制 | ✅ 已确认 |
| BR-003 | 合同签署人必须是系统中拥有`超级签署人`权限的用户。 | 权限规则 | 签署按钮展示与功能 | ✅ 已确认 |
| BR-004 | 合同标题不能为空，合同有效截止日期不能早于开始日期。 | 校验规则 | 新建/编辑操作 | ✅ 已确认 |
| BR-005 | 合同作废后，关联的客户信用额度自动释放。 | 数据联动规则 | 作废/恢复操作 | ❓ 待确认 |

## 5. 数据流 (Data Flow)

- **数据来源**：
  - 页面初始化时，从`/api/contracts` 获取用户有权限查看的合同列表。
  - 创建合同所需的基本客户信息从`/api/customers` 获取。
- **数据变换**：
  - 用户在此页面完成“填写”、“编辑”、“提交”、“签署”等操作，均由前端收集数据，打包成JSON对象，经 `/api/contracts/upsert` 等接口发送到后端。
  - 合同金额字段在前端进行格式化（千分位）展示，存储则保存为数字。
- **数据消费**：
  - 合同一旦“已签署”，其核心信息（客户ID、合同ID、总金额）被`order`模块消费，用于生成销售订单。
  - 合同状态变更（如“已签署”）会触发财务模块的信用额度处理。
- **跨页面数据一致性要求**：
  - `合同-客户`关联：如果客户名在CRM模块变更，本页面的合同列表应同步更新（或通过历史快照保证一致性）。
