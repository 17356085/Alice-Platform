| **RULE-003** | **状态变更影响**：当菜单状态被更改为“禁用”后，所有被分配了该菜单的角色，其用户在下一次登录或会话刷新后将看不到该菜单。 | 数据状态 | 跨页面E2E测试 |
| **RULE-004** | **URL权限绑定**：一个菜单要么是内部组件路由，要么是外部链接。内部路由必须配置正确的 `组件路径` (`component`)，外部链接必须配置完整的 `URL`。 | 数据绑定 | 功能测试、E2E测试 |
| **RULE-005** | **排序规则**：同级菜单按 `排序号` (`sort`) 升序排列。排序号越小，在列表和侧边栏中越靠前。 | 数据展示 | 功能测试 |
| **RULE-006** | **操作按钮必须关联菜单**：一个“按钮”类型的权限点必须挂载在一个“菜单”或“目录”下，不能独立存在。 | 数据关联 | 数据校验 |
| **RULE-007** | **根节点约束**：系统必须存在一个根节点（如“系统管理”），且该节点不能被删除。 | 数据约束 | 异常流程测试 |

## 5. 数据流 (Data Flow)

**数据来源:**
- `新增`、`编辑`操作的数据通过 `POST /system/menu` 或 `PUT /system/menu` API 发送到后端。

**数据变换:**
- `新增`数据由用户输入到前端表单，前端进行必填项、格式等基本校验后，转换成JSON格式发送给后端。
- `编辑`数据由后端查询已有记录，前端表单回显。用户修改后，将修改后的完整JSON发送给后端，进行全字段更新。
- `排序号`在前端提交时作为明文数字传递。
- `状态`（正常/禁用）在前端作为 `el-switch` 或 `el-radio-group` 的值传递。

**数据消费:**
- **<角色管理页面>**: 该页面会加载`菜单管理`配置的树形结构，用于为角色分配权限。其展示的数据直接来源于`菜单管理`的表。
- **<用户侧边栏>**: 用户登录后，后端根据其角色权限，查询`菜单管理`的数据并动态生成侧边栏。菜单的`状态`、`显示顺序`直接影响侧边栏展示效果。
- **<按钮权限>`v-permission`指令**: `菜单管理`中定义的`权限标识`被前端的 `v-permission` 指令消费，用于控制页面内按钮的显示/隐藏。
- **<路由注册>**: 前端路由守卫或动态路由注册表会读取`菜单管理`的 `路由地址` 和 `组件路径` 来注册路由。

**跨页面一致性要求:**
- `菜单管理`中的数据与`角色管理`加载的菜单列表必须实时保持一致。
- `菜单管理`中对菜单的`状态`变更，必须能实时（或在下一次登录时）影响到用户的侧边栏和按钮权限。
