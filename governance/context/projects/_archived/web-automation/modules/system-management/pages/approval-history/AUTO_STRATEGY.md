## 2. PageObject 拆分方案

| Page 类 | 职责 | 包含元素 |
|----------|------|----------|
| `ApprovalHistoryPage` | 我已审批页面的主体操作 | 搜索表单、表格、分页、重置/搜索按钮 |
| `ApprovalDetailDialog` | 查看详情弹窗 | 弹窗标题、详情信息、关闭按钮 |

**拆分依据**：  
- 弹窗作为独立组件，渲染位置在 `<body>` 层级，与主页面 DOM 隔离，独立 Page 类便于维护。  
- 页面主体不再包含弹窗方法，职责单一。

### 可能的子组件（内联在 ApprovalHistoryPage 中）
- `SearchForm` — 搜索表单（但考虑到只有三个字段，直接放在 Page 类中即可，无需额外封装）
- `TableComponent` — 表格操作（同样直接内置）

## 4. 等待策略建议

| 场景 | 异步行为 | 等待策略 |
|------|----------|----------|
| 点击搜索按钮 | 发起请求 → 加载遮罩 → 刷新表格 | 显式等待 Loading 消失后，等待至少一条表格行出现（或“暂无数据”提示） |
| 点击重置按钮 | 清空表单 → 重新加载默认数据 | 同上 |
| 打开详情弹窗 | 请求明细数据 → 弹窗渐入 | 等待 `.el-dialog` 可见，并等待其内部内容非空（或特定标题） |
| 下拉选择 option | el-select 展开/收起有动画 | 使用 `wait_until_option_visible`（已封装） |
| 日期范围选择器 | 弹出日期面板 | 直接点击输入框后等待 `.el-date-picker` 可见 |

**建议封装**：  
在 `ApprovalHistoryPage` 中实现 `_wait_for_table_refresh()` 方法，在搜索、重置后统一调用。

## 5. ROI 分析

### 假设前提
- 手工执行一次完整回归（含 P0+P1 约10个用例）：**15 分钟**
- 手工回归频率：**每日 2 次**（SIT 阶段）
- 自动化开发+调试时间：**8 小时**（1人天）
- 自动化维护成本：**2 小时/月**（定位器变更、UI微调）
- 月度执行频次：**60 次**（30天 × 2次/天）

### 计算（3个月维度）

| 项目 | 手工执行 | 自动执行 |
|------|----------|----------|
| 单次执行时间 | 15 min | 2 min（含 CI 排队） |
| 月度执行时间 | 60 × 15 = 900 min (15h) | 60 × 2 = 120 min (2h) |
| 3个月执行时间 | 45h | 6h |
| 开发成本 | 0 | 8h |
| 维护成本（3个月） | 0 | 6h |
| **总成本** | **45h** | **20h** |

### 结论
- **3个月可收回投资**（自动化总成本20h < 手工45h）。
- 若项目持续超过3个月，自动化收益显著。
- 若仅用于上线前一次性验证，则自动化不划算。

## 6. 风险与注意事项

1. **工厂代码下拉选项动态加载**  
   - 搜索时工厂代码依赖接口数据，若测试环境无数据，选择框为空，导致搜索失败。  
   - **措施**：预置一条测试工厂数据，或通过 API 预置。  
2. **表格数据不稳定**  
   - 第一条数据随时可能变化，导致断言硬编码失败。  
   - **措施**：使用动态断言（如包含特定字段）或通过搜索条件锁定数据。  
3. **详情弹窗层级**  
   - 若存在多层弹窗（如审批详情再次弹窗），需注意 `el-dialog` 的 `wrapper` 是否唯一。  
   - **措施**：使用 `Allure` 截图辅助调试。  
4. **定位器维护**  
   - 当前定位方案基于文字标签（A级稳定），若将来 UI 文字变更，XPATH 需同步调整。  
   - **措施**：定期运行自动化并监控失败率。
