# 业务场景模型：审批链配置模块 (Approval Chain Module)

## 1. 业务目标 (Business Goal)

- **核心业务目标**: 允许系统管理员/流程管理员快速、准确地配置业务单据（如采购合同、费用报销）的审批路径，确保“人找对了，流程不走死”。**一个错误的审批链将直接导致业务单据审批阻塞或通过不合规，构成重大业务风险。**
- **次要业务目标**: 支持灵活的审批规则，如：
    - 按金额/部门/类型等条件动态分配审批人。
    - 支持“会签”（多人都同意）、“或签”（一人同意即可）。
    - 配置超时未处理时的自动提醒或自动转交。
- **非目标**: 本页面不负责创建正在运行的审批实例，也不展示当前审批进度查询。审批的运行时监控在“我的待办”或“流程监控”页面。

## 2. 角色与旅程 (Roles & Journeys)

### 角色列表
| 角色 | 职责边界 | 关键权限 |
|---|---|---|
| **系统管理员 (Admin)** | 拥有所有权限，包括查看、创建、编辑、删除任何审批链。 | 完全控制 |
| **流程管理员 (Workflow Admin)** | 可以管理特定业务范围的审批链（如仅销售部的合同审批），但不能删除与其他业务流程关联的审批链。 | 限制性配置、管理权限 |
| **业务主管 (Business Manager)** | 只读或仅能建议修改，无直接配置权限。 | 查看权限 |

### 典型角色旅程 (以流程管理员为例)

**旅程1: 为“普通采购合同”配置一个新审批链 (Happy Path)**
1. **登录 & 导航**: 登录系统后，通过侧边栏导航到“工作流管理” -> “审批链配置”。
2. **创建**: 点击“新建审批链”按钮，打开配置弹窗。
3. **基本信息配置**: 输入审批链名称“普通采购合同审批链”、描述“用于金额<5万的合同”、选择关联业务类型“采购合同”。
4. **节点1配置**: 点击“添加审批节点”。设置节点名称“部门经理审批”，审批人类型为“用户”，从用户选择列表中选择“张三”。
5. **节点2配置**: 点击“添加审批节点”。设置节点名称“财务经理审批”，审批人类型为“角色”，选择“财务经理”。
6. **条件分支（可选）**: 添加条件：“如果金额 > 2万，则额外增加节点3：财务总监审批”。
7. **保存**: 点击“保存”按钮。系统返回“保存成功”。该审批链出现在列表中。

**旅程2: 修改一个正在被使用的审批链 (Alternative Path 1)**
1. **查看**: 在列表页找到“普通采购合同审批链”。
2. **编辑**: 点击“编辑”按钮。
3. **修改**: 将节点2“财务经理审批”的审批人从“角色：财务经理”改为“用户：李四”。
4. **保存并提示**: 点击“保存”后，后端API检查到此审批链关联了3个未完成流程。系统弹出提示：“此操作将影响3张正在审批的采购单，其中1张已到达节点2。新规则将对新发起的审批生效，已到达节点2的审批将按旧规则完成。确认修改？”
5. **确认**: 管理员点击“确认”。操作成功。系统记录变更日志。

**旅程3: 尝试删除一个被大量引用的审批链 (Alternative Path 2 - 异常处理)**
1. **删除**: 在列表页找到并点击“删除”按钮。
2. **强警告**: 后端返回该审批链被50个活跃流程引用。前端弹出强警告对话框：“此审批链正被50个活跃流程引用，删除将导致它们无法继续。请输入‘确认删除’以完成操作。”
3. **用户输入**: 用户输入“确认删除”，点击“确定”。
4. **逻辑删除**: 后端执行逻辑删除（`is_deleted = True`），所有引用该审批链的流程实例进入“错误/挂起”状态，并触发通知给流程管理员。

## 3. 业务流程 (Workflows)

- **主流程**: 配置审批链 -> 保存 -> (其他页面) 用户发起申请并选择该审批链 -> 流程实例启动 -> 按节点逐级审批 -> 流程结束。
- **子流程 (本页面)**: 创建/编辑 -> 前端校验/后端环路检测 -> 保存 -> 更新列表。
- **分支流程**: 删除审批链 -> 关联检查 -> 强警告 -> 逻辑删除。
- **异常流程**: 保存失败 (网络/后端错误) -> 显示错误信息 -> 用户决定重试或取消。

**流程描述 (Happy Path)**:
1. 用户在审批链列表页看到已有模板。
2. 用户点击“新建审批链”。
3. 输入名称和描述。
4. 用户添加N个审批节点，为每个节点指定审批人（用户/角色/组）。
5. 用户可选地设置条件分支。
6. 用户点击“保存”。
7. 系统校验通过（无环路、无逻辑缺口），显示保存成功。列表页刷新显示新审批链。

**流程描述 (Alt Path 1 - 条件逻辑缺失)**:
1. 用户创建一个金额条件分支：<1000元 -> 张三审批，>5000元 -> 李四审批。
2. 用户点击“保存”。
3. 系统前端/后端校验发现金额范围1000-5000未被覆盖。
4. 系统提示“条件不完整，请添加覆盖1000-5000元区间的审批节点或兜底节点”。
5. 用户添加一个兜底节点，再次保存成功。

## 4. 业务规则 (Business Rules)

| 规则ID | 规则描述 | 层级 | 验证点 |
|---|---|---|---|
| BR-001 | **环路检测规则**: 一个审批链必须能形成一个无环有向图。 | 强制 | 保存时，前端和后端必须检查环路。规则如：链路的终结节点不能回溯到之前的任何节点。 |
| BR-002 | **条件分支完整性规则**: 如果使用了条件分支，所有逻辑（输入类型）的取值集合必须被完整覆盖。 | 强制 | 例如，如果金额是整数，条件 `<1000` 和 `>5000` 需要覆盖 `1000-5000` 的范围，或者必须提供一个“其他情况”的默认分支。 |
| BR-003 | **审批人有效性规则 (运行时)**: 流程启动时，对应的审批人用户必须处于“在职”状态。 | 配置建议 | 配置时应有提示或校验。运行时，系统若找不到在职审批人，应触发通知或自动将流程发给其上级。 |
| BR-004 | **删除引用约束 (强)**: 如果审批链被>0个活跃流程实例引用，应禁止物理删除。逻辑删除后，活跃流程实例将中断。 | 强制 | 删除API必须检查关联的 `running_instances_count`，若>0则拒绝 `DELETE` 请求，只能 `PATCH` 为 `is_deleted=True`。 |
| BR-005 | **审批链名称唯一性**: 在同一业务类型下，审批链名称必须唯一。 | 强制 | 创建/编辑时验证。若已存在则提示“名称已被占用”。 |
| BR-006 | **权限规则**: 非管理员（如流程管理员）只能管理其权限范围（如业务单元）内的审批链。 | 强制 | API需携带用户上下文，过滤返回的审批链列表。 |

## 5. 数据流 (Data Flow)

- **数据来源**: 数据主要在“审批链配置页”创建。
    - **创建**: 用户在页面上**输入**名称、描述、节点、条件等。
    - **引用**: 选择审批人时，数据来源于 `用户管理` 模块的 `/api/users` 接口。
    - **关联业务类型**: 从 `业务配置` 模块的 `/api/business-types` 下拉选择。
- **数据变换**: 前端将配置组装成一个JSON对象（树形/嵌套结构），发送给后端API。
- **数据消费**:
    - **流程启动页**: 当用户发起申请（如“提交采购合同”）时，需要选择“使用的审批链”。此页面从 `审批链管理API` 拉取当前活跃的审批链列表。
    - **流程引擎 (核心)**: 流程实例启动后，引擎会读取此审批链的配置JSON，解释执行每个节点的审批逻辑。
    - **审批页面**: 审批人看到的审批信息（如“当前节点名称”）源自此数据。
- **跨页面数据一致性要求**:
    - 用户管理页修改了用户名字，审批链配置页中已选择的审批人显示的名称不应随之变化（通常显示为历史快照，或实时通过用户ID去查最新信息）。这取决于产品设计，但需求要明确。
    - 用户管理页删除了一个用户，已经配置并包含了该用户的审批链应如何处理？（见风险BRS-003）。

**缺失覆盖的业务场景 (待确认):**
1.  **审批链的“版本控制”**: 当一个审批链被修改后，正在进行的流程实例是使用旧版本还是新版本？这影响“编辑”操作的测试场景设计。
2.  **临时替换审批人**: 当审批人出差时，是否支持管理员在某一个流程实例级别临时修改审批人？这不属于审批链配置页的功能，但通常是流程管理的一部分。
3.  **跨系统审批链配置**: 审批链是否允许配置“跨系统”的审批节点（例如调用外部HR系统审批）？这会导致接口和数据风险显著增加。
