# 测试设计：备件管理页面 (Spare-Item Management)

## 一、概述

本文档基于 `warehouse/spare-item` 页面的风险模型与业务场景，设计系统化的测试方案。目标是确保备件管理模块的功能正确性、业务逻辑完整性、数据一致性与系统健壮性。

## 二、测试维度与策略

### 1. 页面加载与显示
- **目标**: 验证页面在不同状态下的正确渲染。
- **策略**:
    - 正常加载：列表、搜索栏、分页器、操作按钮正常显示。
    - 空数据状态：无匹配数据时，显示“暂无数据”提示，搜索栏、分页器状态正常。
    - 数据加载中：异步加载列表时，显示Loading状态。
    - 权限受限状态：无权限用户登录后，编辑、删除按钮应隐藏或置灰。

### 2. 搜索与筛选
- **目标**: 验证搜索功能的准确性、完整性与容错性。
- **策略**:
    - 精确搜索：按编码、名称精确匹配。
    - 模糊搜索：按名称关键词模糊匹配。
    - 组合搜索：同时使用编码与名称进行搜索。
    - 重置筛选：清空搜索条件后，列表恢复为初始状态。
    - 边界与异常：超长文本、特殊字符、XSS/注入尝试搜索。`(风险ID: RISK-SI-003)`

### 3. 表格操作
- **目标**: 验证数据表格组件的标准交互功能。
- **策略**:
    - 排序：按安全库存、价格等数字字段升/降序排序。
    - 分页：首页、末页、上一页、下一页、跳转指定页码、改变每页显示条数。
    - 行选中：单行/多行选中，全选/全不选。
    - 批量操作：选中多行后，批量编辑/删除等按钮状态变化。

### 4. 新增备件
- **目标**: 验证创建新备件的完整流程，包括输入校验、数据提交、异常处理。
- **策略**:
    - 表单校验：必填项（编码、名称）、格式校验（编码规则 `BR-SI-001`）、唯一性校验（编码重复时提示 `RISK-SI-003`）。
    - 边界值：编码长度边界，库存数量边界（`BR-SI-002`）。
    - 成功/失败提交：提交成功后列表刷新，提交失败后显示错误信息（如网络超时 `RISK-SI-005`）。
    - 数据清理：创建的数据必须在teardown中按测试前缀清理。
    - 并发与异常：重复点击提交按钮、Token过期提交（`RISK-SI-006`）。

### 5. 编辑备件
- **目标**: 验证修改备件信息的正确性、数据回显及冲突处理。
- **策略**:
    - 数据回显：编辑弹窗应正确加载并显示所有现有字段值。
    - 字段修改：合法/非法修改，保存后列表数据更新。
    - 并发编辑：模拟两个用户同时编辑同一条记录，后提交者应获取冲突提示（`RISK-SI-010`）。
    - 权限校验：普通操作员编辑核心字段（如价格）应被拦截（`RISK-SI-002`）。

### 6. 删除备件
- **目标**: 验证删除操作的确认流程、数据状态变更及业务约束。
- **策略**:
    - 流程：单个/批量删除，弹窗确认，取消/确认操作。
    - 业务规则：删除有关联业务单（如入库单）的备件应被禁止（`BR-SI-004`）。
    - 数据安全：执行软删除（`RISK-SI-001`），验证数据库`status`字段或`delete_flag`字段变化。

### 7. 权限测试
- **目标**: 验证不同角色的权限控制是否生效。
- **策略**:
    - **WH_Admin**: 可增、删、改、查。
    - **WH_Supervisor**: 可查看、审核。
    - **Purchaser**: 仅可查看。
    - **API层验证**: 直接调用修改/删除API，检查权限校验是否生效。（`RISK-SI-002`）

### 8. 异常场景
- **目标**: 验证系统在非正常情况下（网络、服务、认证）的容错处理能力。
- **策略**:
    - 网络异常：接口超时或返回`5xx`错误时，前端应有非侵入性错误提示，且页面功能不崩溃（`RISK-SI-005`）。
    - 认证异常：Token过期后操作，前端拦截并引导用户重新登录，同时保留已填信息到草稿（`RISK-SI-006`）。
    - 服务端业务异常：违反`BR-SI-002`规则，后端应返回明确的业务错误码和提示。

### 9. 业务场景验证 (P2-5: 强制)
- **目标**: 验证跨页面、跨角色的完整业务流程，确保业务规则被正确执行。
- **策略**:
    - **BS-SI-001 (管理员管理备件)**: 管理员登录 → 新增 → 搜索 → 编辑 → 删除 → 登出。 Happy Path。
    - **BS-SI-002 (主管审核停用)**: 管理员提交“停用申请” → 主管登录 → 查看待审核列表 → 批准。 多角色协作。
    - **BS-SI-003 (采购员查看信息)**: 采购员登录 → 搜索备件 → 查看详情 → 复制信息。 角色旅程。
    - **BS-SI-004 (低安全库存查询)**: 设置一个备件库存低于安全库存，管理员通过搜索列表定位该备件，触发补货流程。

## 三、自动化优先级建议

| 测试维度 | 手工/自动化 | 优先级 | 说明 |
| :--- | :--- | :--- | :--- |
| 页面加载与显示 | 混合 | P1 | 空数据、loading状态等常见状态适合自动化。权限受限状态可自动化。 |
| 搜索与筛选 | 自动化 | P0 | 高频核心功能，回归成本低，自动化收益高。 |
| 表格操作（排序、分页） | 自动化 | P1 | 基础功能，适合接口+UI验证。 |
| 新增备件 | 自动化 | P0 | 核心流程，必须自动化回归。 |
| 编辑备件/并发 | 自动化 | P1 | 数据回显与并发编辑场景价值高，需自动化。 |
| 删除备件 | 自动化 | P0 | 核心流程，必须自动化。 |
| 权限测试 | 自动化 | P1 | 角色权限是安全基础，建议API+UI覆盖。 |
| **异常场景(网络/超时/Token)** | **不推荐自动化** | - | 网络异常、Token过期的模拟成本高，效果不可控，建议手工或通过`cy.intercept`低代码工具实现。 |
| **端到端业务场景** | **不推荐自动化** | - | 跨页面E2E场景脆弱、不稳定、维护成本高，建议回归时重点关注核心接口与核心UI，E2E保留1-2条Happy Path。 |
| **视觉/布局校验** | **不推荐自动化** | - | el-dialog截断或分辨率适配等问题，更适合手工探索。 |
