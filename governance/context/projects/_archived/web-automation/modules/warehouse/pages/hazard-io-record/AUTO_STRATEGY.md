# 自动化策略：环保出入库明细表

- **模块**: `warehouse`
- **页面**: `hazard-io-record` (HazardIORecordPage)
- **类型**: 只读审计日志列表（无增删改）

---

## 1. 自动化覆盖矩阵

| 用例编号 | 标题                     | 优先级 | 是否自动化 | 理由                                                                 |
|----------|--------------------------|--------|------------|----------------------------------------------------------------------|
| TC‑001   | 页面正常加载（表格渲染）   | P0     | ✅          | 基础冒烟，定位器`TABLE_ROWS`为B级，稳定                               |
| TC‑002   | 分页器可见               | P0     | ✅          | 组件可识别，定位器`TOTAL_COUNT`为B级，稳定                            |
| TC‑003   | 重置搜索                 | P1     | ✅          | 基础功能，但当前测试无断言，建议增强（如检查日期被清空），风险较低     |
| （无）   | 日期范围查询             | P2     | ❌          | 仅覆盖一次查询意义不大；数据动态，断言困难，手工可查                   |

### 风险标注
- 无C级定位器，所有定位器均为B级，风险低。

### 不自动化理由汇总
- 日期范围查询场景：数据高度动态，断言“结果正确”需要强依赖后端预期，成本远高于收益；且只读页面手工执行仅需5秒。

---

## 2. PageObject 拆分方案

当前页面极其简单，无需拆分。建议保持一个 Page 类：

| Page 类 | 职责 |
|---------|------|
| `HazardIORecordPage` | 搜索区操作（日期选择/查询/重置）、表格行读取、分页信息获取 |

**无独立弹窗/对话框**，因此不涉及额外类。

若未来增加“导出”功能（如`el-button`触发下载），可保留在当前类中。

---

## 3. 公共组件复用分析

| 操作                     | 是否可复用 BasePage 方法 | 方法名 / 说明                               |
|--------------------------|--------------------------|---------------------------------------------|
| 导航到页面               | ✅ 由 fixture 封装        | `BasePage.navigate_to()` + 子类 `navigate()` |
| 等待 Vue 稳定            | ✅                       | `BasePage.wait_vue_stable()`                 |
| 等待加载遮罩消失         | ✅                       | `BasePage._wait_loading_gone()`             |
| 点击按钮                 | ✅                       | `BasePage.click()`                           |
| 查找元素                 | ✅                       | `BasePage.find_element()` / `driver.find_elements()` |
| 获取表格行               | ❌ 无需复用               | 直接使用 `driver.find_elements(TABLE_ROWS)` 即可 |
| 日期选择器操作（当前未用）| ❌ BasePage 无对应方法   | 需要扩展 `ElementPlusHelper` 或自行封装      |

**建议**：若后续需自动化日期查询，可在 `BasePage` 或新工具类中加入 `el-date-picker` 的通用操作（如选择最近7天）。

---

## 4. 等待策略建议

| 场景                     | 等待条件                                    | 建议封装                         |
|--------------------------|---------------------------------------------|----------------------------------|
| 页面加载完成             | 表格至少有一行（或空数据占位）且加载遮罩消失 | `_wait_page_ready()` 已存在 ✅    |
| 点击查询/重置后          | 加载遮罩消失 + Vue 稳定                      | `_wait_loading_gone()` + `wait_vue_stable()` |
| 分页切换（若未来需要）   | 表格行重新渲染                              | 同上，无需额外等待               |

**注意事项**：
- 日期选择器下拉面板（`el-popper`）是瞬态元素，需显式等待 `visibility_of_element_located`，当前测试未涉及。
- 页面无异步接口调用（数据可能一次性加载），但建议保留 `wait_vue_stable()` 确保 Vue 响应完成。

---

## 5. ROI 分析

| 项目               | 估计值              | 计算说明                                       |
|--------------------|---------------------|-------------------------------------------------|
| 自动化开发时间     | 2 小时              | 已有 Page Object + 测试脚本，仅需微调和增强断言 |
| 月维护成本         | 0.5 小时/月         | 定位器极少变动，主要维护时间用在框架升级或环境问题 |
| 手工单次执行时间   | 2 分钟              | 点击页面+查看结果                               |
| 手工执行频率       | 1 次/天 (20天/月)   | 假设每日冒烟执行                                |
| **年度 ROI**       | **≈ + 32 小时**     | 计算: (2min × 20 × 12 - 0.5h × 12 - 2h) / 60 ≈ 32 小时节省 |

**结论**：自动化覆盖基础冒烟（2个P0 + 1个P1）价值明确，开发成本低，维护成本极低，建议立即实施。日期查询场景不纳入自动化（ROI 为负）。
