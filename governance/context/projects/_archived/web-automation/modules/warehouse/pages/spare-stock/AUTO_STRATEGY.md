## 2. PageObject 拆分方案

根据"一个页面一个Page类"的原则，建议如下拆分：

```
page_objects/
└── warehouse/
    └── spare_stock/
        ├── __init__.py
        └── spare_stock_page.py          # 主页面类（无弹窗，单一文件）
```

### `SpareStockPage`
- **职责**: 封装库存查询页面的搜索、表格、分页操作。
- **核心方法**:
  - `search_by_item_name(name: str)` -> `self`
  - `click_reset()` -> `self` (或 `reset_search()`)
  - `get_total_count()` -> `int`
  - `check_page_loaded()` -> `bool`
- **不包含**: 无 CRUD 方法，无弹窗处理。

## 3. 公共组件复用分析

### 3.1 复用 `BasePage` 已有方法（高频）

- **`wait_for_element_visible(locator)`**: 搜索框、按钮、表格交互前调用。
- **`click_element(locator)`**: 点击查询、重置按钮。
- **`type_text(locator, value)`**: 搜索区的物品名称输入框。
- **`get_element_text(locator)`**: 获取分页总条数。
- **`take_screenshot(name)`**: 测试失败时截图。

### 3.2 复用 `ElementPlusHelper`（低频）

- 本页面搜索区仅有 `el-input` 文本输入框，无需使用 `el-select`、`el-date-picker` 等复杂组件。`ElementPlusHelper` 仅在通用等待和表格交互中使用。

### 3.3 不需要新建工具方法

- 本页面无弹窗、无复杂交互，现有 `BasePage` 方法已可完全覆盖。

## 4. 等待策略建议

该页面特有的异步行为建议如下处理：

1. **表格加载**:
   - 行为: 搜索/重置/分页切换后，表格数据异步请求并渲染。
   - 策略: 等待 `el-table` 中的 loading 遮罩消失，或等待表格行出现。

2. **搜索**:
   - 行为: 输入物品名称后点击查询，页面异步请求。
   - 策略: 输入内容后点击查询按钮，等待 loading 消失，然后检查列表数据。

3. **重置**:
   - 行为: 点击重置后，搜索框内容清空，表格恢复初始数据。
   - 策略: 点击重置按钮，等待 loading 消失，验证搜索框值为空。

```python
# 建议的等待封装结构
def wait_for_table_ready(page, timeout=10):
    """等待表格数据加载完毕"""
    page.wait_for_element_invisible(by_css='.el-loading-mask')
```

## 5. ROI 分析

### 假设条件

- 预期执行频率: **每日 3 次** (CI/CD构建)。
- 手工执行时间: 完成页面加载、搜索、重置、分页测试约为 **8 分钟/次**。
- 预估自动化开发时间: **2 小时** (仅 `SpareStockPage` 和 4 个测试方法)。
- 预估月度维护成本（按平均每月页面变动2次计算）：**0.3 小时/月**。

### ROI 计算

- **月度手工执行时间**: `8 分钟/次 × 3 次/天 × 22 工作日/月` = **528 分钟/月** ≈ **8.8 小时/月**
- **月度自动化执行时间**: 假设自动化脚本执行耗时 1 分钟，则 `1 分钟/次 × 3 次/天 × 22 天/月` = **66 分钟/月** ≈ **1.1 小时/月**
- **月度节省时间**: `8.8 - 1.1 = 7.7 小时/月`
- **月度净收益**: `月度节省时间 - 月度维护成本` = `7.7 - 0.3` = **7.4 小时/月**
- **投资回收期**: `总投资（开发时间） / 月度净收益` = `2 / 7.4` ≈ **0.27 个月 → 约1周**

**结论**: 自动化测试的投资回报率（ROI）较高。初始开发的 **2 小时** 成本，在 **约1周** 内即可收回，后续每月可释放约 **7.4 小时** 的人力资源。由于页面无弹窗和 CRUD 操作，开发成本较低，ROI 表现良好。
