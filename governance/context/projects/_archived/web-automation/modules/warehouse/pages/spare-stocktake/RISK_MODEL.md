# 风险模型: warehouse → spare-stocktake

## 页面概述
- **页面名称**: 备品库存盘点 (Spare Stocktake Query Page)
- **模块**: 仓库管理 (Warehouse)
- **核心功能**: 展示备品盘点记录，支持按盘点人和日期搜索查看。只读查询页面，无 CRUD 操作。盘点记录有审批流（chenqian → tjw）。

## 风险识别

### 1. 业务风险 (Business Risks)

| 风险ID | 风险描述 | 等级 | 缓解措施 |
|--------|----------|------|----------|
| RISK-ST-001 | **盘点记录未更新**: 用户发起的盘点操作已完成，但查询页面未及时展示最新的盘点记录，导致业务人员误以为盘点未完成。 | P1 | 1. 盘点完成后强制刷新查询页面。<br>2. 页面增加"最后更新时间"标签。 |
| RISK-ST-002 | **审批状态不准确**: 盘点记录的实际审批状态与页面显示不一致，导致审批流程混乱或重复审批。 | P0 | 1. 审批状态应实时从审批流服务获取。<br>2. 前端定期轮询或通过 WebSocket 推送状态变更。 |
| RISK-ST-003 | **盘点结果查看权限不当**: 盘点结果包含库存差异明细，非授权人员可能看到不应看到的差异数据。 | P1 | 1. 盘点明细的查看权限应独立控制。<br>2. 根据角色动态展示/隐藏明细信息。 |
| RISK-ST-004 | **搜索不准确**: 按盘点人模糊搜索匹配不完整，或日期范围选择后数据不准确，导致用户无法找到目标盘点记录。 | P2 | 1. 支持模糊搜索和精确搜索。<br>2. 日期筛选前后端时区一致。 |

### 2. 权限风险 (Permission Risks)

| 风险ID | 风险描述 | 等级 | 缓解措施 |
|--------|----------|------|----------|
| RISK-ST-005 | **越权查看盘点数据**: 普通用户通过 URL 直接访问盘点查询页面，看到不应看到的盘点信息。 | P0 | 1. 后端 API 必须校验用户对盘点数据的访问权限。<br>2. 不能仅依赖前端菜单隐藏。 |
| RISK-ST-006 | **审批流参与人越权**: 审批链 chenqian → tjw 之外的用户不应看到待审批的任务，但页面可能显示了全部盘点记录。 | P0 | 1. 后端根据用户角色过滤审批任务。<br>2. 页面默认加载用户有权限的数据范围。 |

### 3. 数据风险 (Data Risks)

| 风险ID | 风险描述 | 等级 | 缓解措施 |
|--------|----------|------|----------|
| RISK-ST-007 | **搜索框输入注入攻击**: 盘点人搜索框输入特殊字符导致 SQL 注入或 XSS 攻击。 | P0 | 1. 使用参数化查询。<br>2. 对展示字段进行 HTML 转义。 |
| RISK-ST-008 | **日期范围选择异常**: 跨月/跨年日期范围选择后，后端查询结果与预期不一致（如时区转换错误、日期边界问题）。 | P1 | 1. 前后端使用统一时间格式（如 UTC）。<br>2. 测试跨月、跨年、月末等边界情况。 |
| RISK-ST-009 | **空列表展示不友好**: 搜索无匹配结果后，页面仅显示空白区域。 | P2 | 提供明确的空状态提示和操作建议。 |

### 4. 接口风险 (Interface Risks)

| 风险ID | 风险描述 | 等级 | 缓解措施 |
|--------|----------|------|----------|
| RISK-ST-010 | **接口超时加载态不清晰**: 查询盘点列表时接口响应慢，页面一直显示 Loading 遮罩。 | P1 | 1. 设置前端请求超时时间，超时后展示重试按钮。<br>2. 请求期间阻止用户反复点击搜索按钮。 |
| RISK-ST-011 | **分页接口返回异常**: 请求第 N 页，后端返回了错误页的数据。 | P1 | 前端对返回的分页数据结构进行校验。 |
| RISK-ST-012 | **Token 过期无感知**: Token 过期后接口返回 401，页面无跳转或无明确提示。 | P0 | 前端全局拦截器捕获 401 状态码，自动跳转至登录页。 |

### 5. UI/UX 风险 (UI/UX Risks)

| 风险ID | 风险描述 | 等级 | 缓解措施 |
|--------|----------|------|----------|
| RISK-ST-013 | **表格固定列重叠**: 小屏幕分辨率下 `el-table` 固定列与普通列重叠。 | P1 | 进行多分辨率下的布局测试。 |
| RISK-ST-014 | **日期选择器 Teleport 渲染错位**: `el-date-picker` 日历面板通过 Teleport 渲染到 `<body>` 下，可能出现位置偏移或遮挡。 | P1 | 确认日历面板的 `popper-class` 样式和 `placement` 属性。 |

### 6. 性能风险 (Performance Risks)

| 风险ID | 风险描述 | 等级 | 缓解措施 |
|--------|----------|------|----------|
| RISK-ST-015 | **盘点记录数据量大**: 长期积累的盘点记录导致列表首次加载缓慢。 | P0 | 1. 必须实现后端分页。<br>2. 默认显示前 20 条，翻页加载更多。 |
| RISK-ST-016 | **复杂日期搜索性能**: 跨长时间范围的日期搜索导致后端查询性能瓶颈。 | P2 | 1. 限制最大可查询日期范围。<br>2. 日期字段建立数据库索引。 |

## 业务场景

### 1. 角色与旅程

| 角色 | 职责边界 | 典型旅程 | 置信度 |
|------|----------|----------|--------|
| **仓库管理员 (chenqian)** | 发起盘点，查看盘点记录，提审批。 | 打开页面 → 搜索自己的盘点记录 → 查看盘点结果 → 提交审批至 tjw。 | ⚠️ 推断 |
| **审批人 (tjw)** | 审批盘点记录。 | 打开页面 → 查找待审批记录 → 查看盘点明细 → 审批通过/驳回。 | ⚠️ 推断 |
| **普通仓库人员** | 查看盘点记录，无权审批。 | 打开页面 → 搜索盘点人 → 查看盘点结果。 | ⚠️ 推断 |

### 2. 业务流程

- **Happy Path**: 仓库管理员完成盘点 → 在页面搜索盘点记录 → 确认数据正确 → 提交审批。
- **审批路径**: 盘点人 chenqian 提交流程 → 审批人 tjw 登录 → 搜索/查找待审批记录 → 查看详情 → 审批通过。
- **Alternative Path**: 用户搜索不存在的盘点人 → 页面空状态 → 修改条件重新搜索 → 找到目标记录。
- **异常路径**: 用户选择错误的日期范围 → 未找到目标记录 → 调整日期范围 → 再次搜索。

### 3. 业务规则

| 规则ID | 规则描述 | 验证点 | 置信度 |
|--------|----------|--------|--------|
| BR-ST-001 | **审批链固定**: 盘点审批必须由 chenqian 发起，经 tjw 审批。 | 验证审批流参与者配置。 | ⚠️ 推断 |
| BR-ST-002 | **数据范围控制**: 用户只能查看其有权限的组织/仓库的盘点数据。 | 使用不同角色登录，查看列表数据范围。 | ⚠️ 推断 |

## 未覆盖的高风险场景（自动化用例缺失）
- **RISK-ST-005 / RISK-ST-006**: 缺少权限校验的自动化测试（越权查看盘点数据、审批流访问控制）。
- **RISK-ST-007**: 缺少对搜索框进行 XSS/SQL 注入攻击的接口安全测试。
- **RISK-ST-008**: 缺少日期选择器边界值测试（跨月、跨年）。
- **RISK-ST-015**: 缺少对大数据量下的渲染性能自动化验证。
