## Phase 2: BUSINESS_SCENARIOS.md — 业务场景模型

### 1. 业务目标 (Business Goal)
- **核心业务目标**：为质检员和分析工程师提供 **按取样位置、按时间范围** 快速查看和追溯气体组分分析数据的视图。
- **次要业务目标**：
  1. 支持新增/编辑气体分析报告单，记录原始检测数据。
  2. 支持导出当前视图数据，用于离线分析、报告或存档。
- **非目标**：
  1. 本页面不负责自动采集设备的实时数据（数据由其他系统或人工录入）。
  2. 本页面不进行数据的复杂统计分析（如趋势图，理想状态下应由后台 BI 系统完成）。

### 2. 角色与旅程 (Roles & Journeys)
- **角色一：质检员**
  - **职责边界**：主要职责是报告单的“读”和“录入”。需要查看历史数据，并按需求在特定位置下新增一份完整的分析报告。
  - **典型旅程**：
    1. **查看特定区域最新数据**：`进入页面 → 点击“LNG储罐”Tab → 查看表格最新的一条组分数据（H2, CH4等） → 结束`。
    2. **录入新分析数据**：`进入页面 → 点击“新增”按钮 → 填写报告（取样时间、位置、各个组分的检测值） → 提交 → 页面自动刷新，验证新记录出现在列表顶部`。
- **角色二：质检档案管理员**
  - **职责边界**：关注数据完整性和交叉验证，会进行深度检索、比对数据并导出报表。
  - **典型旅程**：
    1. **验证月度数据完整性**：`进入页面 → 选择“近一个月”日期范围 → 查看所有取样位置的报告数量 → 抽样选择“界区原料气”Tab，核对关键参数无误 → 结束`。
    2. **导出数据制作报告**：`进入页面 → 选择“界区原料气”Tab → 设置起止时间（例如上个月全月） → 点击查询 → 点击“导出” → 系统弹出下载提示对话框 → 点击保存`。

### 3. 业务流程 (Workflows)
- **主流程**：`导航进入` → `按取样位置Tab定位视图` → `按日期筛选` → `浏览与核对数据`。
- **分支流程**：
  - **新增报告**：`浏览中` → `点击“新增”` → `填写表单` → `提交` → `返回当前列表，验证新数据已刷新`。
  - **导出报告**：`筛选后` → `点击“导出”` → `确认文件生成`。
- **异常中断及恢复流程**：
  1. **网络超时**：在进行“查询”操作时网络中断。
     - *预期*：页面应显示 loading 状态，超时后提示“出错了，请稍后重试”。用户可以点击“重置”或再次查询来恢复。
  2. **新增报告失败**：提交新增报告单时服务器返回校验错误（如：必填项为空）。
     - *预期*：页面不应直接跳转，应停留在当前表单，并高亮错误项。用户修正后重新提交。

### 4. 业务规则 (Business Rules)
- **规则 1**：一个“取样位置”+“取样时间”（精确到分钟）的组合应唯一确定一份报告单。系统不能有两份完全相同的报告。
- **规则 2**：所有组分数据（甲烷、乙烷等）在展示时应保留 3 位小数（如 `98.352`）。`0` 必须显示为 `0.000`。
- **规则 3**：当某组分未检测时，该单元格必须显示为 `"-"` （短横线，代表N/A），不能为空字符串或 `null`。
- **规则 4**：分组展示：每组取样位置最多展示最近 30 条数据记录。超过部分需要分页。

### 5. 数据流 (Data Flow)
- **数据来源**：
  - **页面加载**：数据来源于 `GET /api/lab/gas/reports?location_id={id}&start_date={}&end_date={}`。参数由页面上的 Tab 选择和日期输入框决定。
  - **数据新建**：数据通过 `POST /api/lab/gas/reports` 提交。数据在新增报告单表单页面录入。
- **数据变换**：
  - 本页面是纯展示和筛选，不涉及数据变换。所有计算（如平均值）由后端完成，前端仅负责展示。
- **数据消费**：
  - **导出**：数据通过 `GET /api/lab/gas/reports/export?location_id={id}&start_date={}&end_date={}` 接口被消费，生成 Excel 文件。
  - **下游页面**：该页面的报告单可能在“报告审核”模块被审核员调阅。

### 结论

本次风险建模共识别出 **12 个风险点**，其中 **1 个 P0** 和 **7 个 P1**。现有自动化测试覆盖了最基础的展示和主流程，但**对“数据一致性”、“权限安全”、“异常处理”和“性能”等高危区域的覆盖严重不足**。P0 风险的缓解措施需要在 `click_location_tab` 方法中增加数据断言，这是模型给出的最紧急的改进指令。

同时，`BUSINESS_SCENARIOS.md` 明确指出了 **10 个具体可验证的业务场景**，并标记了 **2 个目前缺失覆盖的关键场景**（统计行验证、搜索后重置），为后续的 `test_design` 提供了清晰的输入。
