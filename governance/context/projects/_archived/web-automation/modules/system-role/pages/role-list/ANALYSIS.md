## 1. 需求理解

> **一句话**：角色管理页面是 RBAC 权限体系的**配置中枢**——通过 CRUD 管理 19 个系统角色，并通过「权限」弹窗（500+ checkbox）和「分配用户」弹窗（152 用户穿梭框）完成角色⇔权限⇔用户的**三维映射**，是一切多角色切换功能的数据源。

核心能力分解：
- **角色 CRUD**：创建/编辑/删除角色，管理角色名称、编码、排序、状态
- **权限分配（PC端）**：95 个 perm-group、500+ 权限 checkbox，按一级菜单分类
- **权限分配（小程序端）**：独立的小程序操作权限 Tab
- **数据范围设置**：全部/自定义/本部门及下级/本组织 四种粒度
- **分配用户**：角色→用户的反向关联，与用户管理的「分配角色」形成双向通道

### 2.2 隐式规则（需从上下文推导）

| 编号 | 规则 | 推导依据 |
|:---:|------|------|
| **BR-14** | **系统内置角色（super_admin、admin）不可删除** | §5.3 列出的 19 个角色中有 super_admin 和 admin；RBAC_TEST_PLAN.md TC-RBAC-X04 明确验证此规则 |
| **BR-15** | **权限变更后需要「清空缓存」才生效** | §4 TBR-04：工具栏有「清空缓存」danger 按钮 → 权限变更不是即时生效的，依赖缓存刷新 |
| **BR-16** | **权限弹窗的「未选」和「已选」Tab 是筛选标签，不是独立权限域** | §8.2：tab-1/tab-2 只筛选已勾选/未勾选的节点，真正权限内容在 tab-operations/miniappOperations/dataScope 中 |
| **BR-17** | **分配用户弹窗不是"提交即替换"模式，而是"勾选即加入"模式** | §9.5："左侧表格勾选用户 → 用户自动加入右侧已选用户" → 用户选择是即时反馈的，不需要手动点击"添加"按钮 |
| **BR-18** | **角色的数据范围影响该角色下所有用户的数据可见性** | §8.6：数据权限 Tab 设置的是角色级别的数据范围；MODULE_CONTEXT.md §8.2 说明用户通过角色间接获得数据权限 |
| **BR-19** | **权限 checkbox 按 perm-group 分组，需要展开才能看到子权限** | §8.3 DOM 结构中有 `perm-group__arrow` 展开/折叠箭头图标 → 默认可能是折叠状态 |
| **BR-20** | **权限保存是整体提交（PUT 全量权限），不是增量修改** | §附录A：`PUT /api/system/role/permission/{roleId}` — 无增量 API，每次保存覆盖全部权限 |
| **BR-21** | **管理员可能无法删除/操作 super_admin 角色** | RBAC_TEST_PLAN §7.2 提到 admin 使用 `ensure_logged_in` → super_admin 权限可能高于 admin |
| **BR-22** | **分配用户弹窗批量操作上限未知** | §9.3 左侧共 152 条用户 → 理论上可全选 152 人，但后端可能有批量限制 |

## 3. 测试范围界定

### ✅ 纳入测试

| 类别 | 具体内容 | 理由 |
|------|------|------|
| **角色 CRUD** | 新增（含必填校验、唯一性校验、状态选择、排序生效）<br>编辑（名称/编码/排序/状态/备注修改）<br>删除（单条删除+确认框）<br>勾选行联动（修改/删除按钮 disabled↔enabled） | 角色管理的核心功能，是权限分配的前置条件 |
| **权限分配弹窗** | 打开/关闭弹窗<br>PC操作权限 Tab：展开/折叠 perm-group、勾选/取消 checkbox<br>小程序操作权限 Tab：同上<br>数据权限 Tab：单选切换<br>未选/已选筛选 Tab：正确筛选<br>保存权限 → Toast → 清空缓存<br>权限持久化验证（关闭弹窗重新打开，权限应保留） | 权限矩阵的核心配置入口，直接影响全系统 |
| **分配用户弹窗** | 打开/关闭弹窗<br>左侧面板：搜索用户、部门筛选、分页翻页、勾选用户<br>右侧面板：已选展示、清空功能<br>保存分配 → Toast<br>双向一致性验证（与用户管理的「分配角色」对应） | 角色→用户映射的执行入口 |
| **搜索/筛选** | 按角色名称模糊搜索<br>按角色编码模糊搜索<br>按状态筛选（全部/启用/停用）<br>多条件 AND 组合查询<br>空条件查询显示全部<br>重置按钮清空条件 | 列表页基本功能 |
| **分页** | 翻页切换、每页条数切换 | 列表页基本功能 |
| **权限即时生效** | 修改角色权限 → 清空缓存 → 持有该角色的用户刷新页面 → 权限生效 | 见 BR-15，缓存清空是关键步骤 |
| **内置角色保护** | 删除 super_admin/admin 角色 → 应被拦截 | 见 BR-14 |
| **角色编码唯一性** | 同一编码创建两次 → 提示"角色编码已存在" | BR-02 |

### ❌ 不纳入测试

| 类别 | 具体内容 | 理由 |
|------|------|------|
| **小程序端权限的实际生效** | 小程序操作权限 Tab 保存后在微信小程序端的实际表现 | 不在 Web 管理后台范围内，属于小程序端专项测试 |
| **全部 500+ 权限 checkbox 逐一验证** | 逐个勾选/取消每个 checkbox 验证权限生效 | 工作量过大且 ROI 低；应抽样关键权限组合 + 等价类划分 |
| **与 SAP 系统的权限同步** | 角色权限变更是否同步至 SAP | SAP 接口不在当前测试范围内 |
| **工作流审批链中角色绑定的端到端验证** | 角色绑定审批节点后审批流是否正常流转 | 属于工作流管理模块的测试范围 |
| **权限弹窗的性能/压力测试** | 500+ checkbox 的渲染性能、勾选全部 checkbox 的提交耗时 | 性能测试与当前功能测试范围不同 |
| **并发角色分配的用户端即时体验** | 两个管理员同时对同一角色修改权限、分配用户的并发场景 | 并发测试优先级低于功能正确性测试 |

## 4. 风险识别

### 4.1 需求模糊点

| 编号 | 模糊点 | 影响 | 建议 |
|:---:|------|------|------|
| **AMB-01** | **权限保存是覆盖式还是增量式？** 文档未明确。如果用户打开弹窗时只改了一个 checkbox，保存后是否会覆盖其他管理员对该角色权限的并发修改？ | 并发权限分配可能导致权限丢失 | 确认后端 `PUT /api/system/role/permission/{roleId}` 的全量覆盖语义 → 如果有并发风险，需加乐观锁(version字段) |
| **AMB-02** | **「清空缓存」到底清空了什么？** 只清空权限缓存？还是连用户-角色关联缓存一起清？ | 影响测试验证步骤——如果是全量缓存清空，会影响所有在线用户 | 在测试中验证：1)仅修改权限→清缓存→验证；2)修改角色分配→不清缓存→验证是否生效 |
| **AMB-03** | **分配用户弹窗中「勾选用户→自动加入右侧」的同步机制**：是前端即时移动 DOM 还是调用后端 API？ | 弹窗关闭不点确定时，已勾选的用户状态是否已持久化？ | 测试：勾选用户→关闭弹窗（不点确定）→重新打开分配用户弹窗→检查勾选状态是否保留 |
| **AMB-04** | **「未选(X)」和「已选(Y)」Tab 的计数范围**：是仅统计当前展开的 perm-group，还是全局 500+？ | 如果是全局统计但 DOM 未渲染不可见节点，性能可能有问题 | 用 JS 提取实际计数，对比全局 checkbox 数量 |
| **AMB-05** | **角色删除的前置条件**：文档未明确是否校验「有用户持有该角色时不可删除」 | 带用户的角色被删除后，用户权限断裂 | 测试：给角色分配用户→尝试删除角色→观察行为 |

### 4.2 逻辑矛盾/遗漏

| 编号 | 矛盾/遗漏 | 证据 | 建议 |
|:---:|------|------|------|
| **GAP-01** | **权限弹窗缺少"全选/全不选"功能** | §8.4 描述 500 个 checkbox 但没有提及批量操作按钮。如果管理员想给某个角色分配"系统管理全部权限"，需要手动勾选几十个 checkbox。 | 确认是否有 perm-group 级别的全选功能（勾选分类名称=勾选所有子权限）；如果没有，这是一个 **UX 风险**，误操作概率高 |
| **GAP-02** | **数据权限 Tab 未完整展开** | §8.6 标注"实测中切换到数据权限 Tab 后 active pane 未正确识别（需更长的渲染等待）"→ **数据权限 Tab 的渲染时序可能是 bug** | 在测试中重点验证 Tab 切换的异步渲染时序 |
| **GAP-03** | **权限变更缺少变更日志/审计** | 文档中权限变更后仅展示 Toast"操作成功"，未提及任何变更记录展示 | 操作日志（oper_log）模块应覆盖"权限变更"事件 → 验证操作日志是否记录了"谁、什么时间、修改了哪个角色的权限" |
| **GAP-04** | **分配用户弹窗缺少"全选当前页/全选全部"功能** | §9.2 布局结构中左侧面板没有全选按钮 → 要给一个角色分配 50 个用户需要手动勾选 50 次 | 确认是否有批量选择功能；这是一个可用性问题 |
| **GAP-05** | **工具栏「修改」「删除」与行内「修改」「删除」功能重复** | §4 工具栏有修改/删除按钮（需先勾选行），§5.2 行内也有修改/删除按钮 → **两套操作路径是否等价？** | 测试两种路径（工具栏 vs 行内）的行为一致性 |
| **GAP-06** | **角色编码的修改限制未明确** | §7.2 编辑弹窗中有「角色编码」字段，但 §5.3 显示编码如 `super_admin`、`admin` 是系统预设的。编辑时角色编码是否可修改？修改后是否影响已有权限关联？ | 测试：编辑角色→修改编码→保存→验证角色编码变更后权限和用户关联是否保留 |

### 4.3 与技术架构的冲突

| 编号 | 冲突 | 分析 | 建议 |
|:---:|------|------|------|
| **TECH-01** | **perm-group 组件 vs Element Plus el-tree 的定位差异** | §8.3 明确标注使用自定义 `perm-group` 组件而非 el-tree。现有的 `RoleManagePage.py` 定位器可能基于 el-tree 编写，导致自动化失效。 | 见 ROLE_CONTEXT §12 D-01："PO 预期使用 el-tree-node 定位，实测使用自定义 perm-group" → 这是一个 **已知的 PO 差异**，需要在自动化脚本中修复 |
| **TECH-02** | **权限弹窗的 500+ checkbox 在 Selenium 中的渲染性能** | Element Plus checkbox 组件在大量 DOM 节点时渲染耗时较长。如果所有 perm-group 默认展开，页面 DOM 节点可能达 2000+。 | 确认 perm-group 是否默认折叠 → 如果是，自动化需先展开目标 perm-group 才能操作子 checkbox |
| **TECH-03** | **JWT Token 在权限分配/缓存清空后的状态** | 清空缓存后，已登录用户的 token 中携带的 `permissions[]` 是否自动失效？还是需要刷新 token？ | 与 BR-15 相关 → 测试清空缓存后：已登录用户刷新页面 → 侧边栏菜单是否更新 |
| **TECH-04** | **动态路由的实时性** | §PROJECT_CONTEXT 中动态菜单加载流程：前端递归过滤 + Vue Router.addRoute()。如果用户在线时其角色权限被变更，前端是否需要手动重新触发路由过滤？ | 验证：admin 修改用户角色→用户刷新页面→路由是否正确更新 |

## 5. 测试策略建议

### 5.1 策略选择：**分层混合策略**

```
┌──────────────────────────────────────────────────────────┐
│  Layer 1: API 接口测试 (pytest + requests)      占比: 30% │
│  ├── 覆盖: 角色CRUD 9个API + 权限分配2个API + 用户分配2个API │
│  ├── 优势: 执行快(秒级)，不受UI时序影响，可验证数据完整性        │
│  └── 劣势: 无法验证前端权限树的渲染/交互                      │
├──────────────────────────────────────────────────────────┤
│  Layer 2: UI 自动化 (Selenium + Page Object)    占比: 50%  │
│  ├── 覆盖: 列表页操作 + 弹窗交互 + 权限树操作 + 分配用户操作   │
│  ├── 优势: 验证真实用户视角，覆盖前端渲染/时序/定位差异         │
│  └── 劣势: 执行慢(分钟级)，受 Element Plus 动态渲染影响        │
├──────────────────────────────────────────────────────────┤
│  Layer 3: 手工探索测试                          占比: 20%  │
│  ├── 覆盖: 500+ checkbox 边界、并发场景、异常路径              │
│  ├── 优势: 灵活，适合一次性/低频率探索                       │
│  └── 劣势: 耗人力，不可重复                                 │
└──────────────────────────────────────────────────────────┘
```

### 5.2 为什么是混合而非纯自动化？

| 原因 | 说明 |
|------|------|
| **perm-group 组件定位不稳定** | 自定义组件的 DOM 结构可能在版本迭代中变化，纯自动化维护成本高 |
| **500+ checkbox 全覆盖不可行** | 全量遍历 500 个 checkbox 在 Selenium 中极慢且不稳定，需用等价类 + 抽样策略 |
| **异步渲染时序差异** | 不同网络环境下弹窗渲染时序不同（§GAP-02 已暴露此问题），需要人工判断 |
| **缓存行为的不可预测性** | 清空缓存后权限生效的准确时机依赖后端实现，手工探索可以捕获边缘情况 |
| **已有自动化基建良好** | 15 个用例 + 完整 PO + RBAC 种子脚本 → 自动化成本低，优先自动化确定性高的场景 |

## 7. 自动化机会

### 7.1 适合自动化（高 ROI）

| 功能点 | 优先级 | 自动化方式 | ROI 理由 | 已有基础 |
|------|:---:|------|------|:---:|
| **角色 CRUD 闭环** (新增→搜索→编辑→删除) | P0 | Selenium | 操作稳定，定位器成熟，每次回归必跑 | ✅ 15 条用例已存在，6 条待修复 |
| **权限弹窗保存+验证** (勾选→保存→重新打开验证) | P0 | Selenium + JS | 验证权限持久化是核心路径；JS 可直接读取 checkbox 状态跳过渲染等待 | ✅ RoleManagePage 已封装方法 |
| **分配用户弹窗保存+验证** | P0 | Selenium | 角色⇔用户双向关联的核心验证 | ✅ 已封装 `assign_user_select_first_row_and_confirm` |
| **API 层接口测试** (13 个角色相关 API) | P1 | pytest + requests | 执行快(秒级)，不受 UI 时序影响，可在 CI 中每次提交执行 | ❌ 无现有基础，需新建 |
| **搜索/筛选/分页** (6 个组合) | P1 | Selenium | 标准列表页操作，模式固定，稳定性高 | ✅ 已有用例 TC-SYS-007~011 |
| **权限变更→即时生效** (端到端) | P1 | Selenium + 多用户切换 | 验证 BR-15 缓存逻辑的关键链路 | ❌ 需新建（可复用 rbac_seed 脚本） |
| **内置角色保护** (删除拦截) | P2 | Selenium | 一次验证长期有效，低频率回归 | ❌ 需新建 |

### 7.2 不适合自动化（低 ROI）

| 功能点 | 原因 | 替代方案 |
|------|------|------|
| **500+ checkbox 全量遍历** | 执行耗时（每 checkbox ~1s = 500s），perm-group 展开/折叠定位不稳定 | 等价类抽样：每模块勾选 1 个 + 边界(0个/全模块/单个) → 共 ~30 个场景 |
| **并发角色分配** | Selenium 不支持真正的并发操作 | 手工 + API 层并发测试（Locust/k6） |
| **权限弹窗渲染性能** | 性能指标依赖环境因素，不宜在功能测试中验证 | 手工记录基线 + Chrome DevTools Performance |
| **小程序操作权限 Tab 在小程序端的实际生效** | 跨端验证需要 Appium/小程序自动化框架 | 手工验证 + 截图对比 |

## 总结：角色管理页面的测试优先级矩阵

| 维度 | P0（阻塞发布） | P1（每次回归） | P2（大版本前） |
|------|------|------|------|
| **CRUD** | 新增/编辑/删除主要路径、必填/唯一性校验 | 排序生效、状态切换 | 备注字段、超长输入 |
| **权限弹窗** | 权限保存+持久化+即时生效 | perm-group 展开/折叠、Tab 切换、未选/已选筛选 | 500+ checkbox 全选提交、小程序权限 Tab |
| **分配用户** | 分配保存+用户管理侧联动 | 搜索/筛选/分页、清空功能 | 超大量用户分配、部门筛选 |
| **缓存** | 清空缓存后权限即时生效 | 不清缓存权限不变 | 并发缓存刷新 |
| **内置角色** | super_admin 不可删除 | admin 不可删除 | 其他系统角色保护级别 |
