## 自动化建议

| 场景类型 | 自动化类型 | 示例 | 原因 |
|----------|------------|------|------|
| UI 加载 / 基础展示 | **✅ 适合自动化** | UI-01, TAB-01 | 无状态，稳定，可快速验证 |
| 搜索筛选 / 重置 | **✅ 适合自动化** | SEA-01 ~ SEA-06 | 输入驱动的场景，适合数据驱动自动测试 |
| 分页 | **✅ 适合自动化** | TAB-02, TAB-03 | 逻辑明确，适合参数化测试 |
| 审批操作（通过/驳回） | **🔄 混合** | BS-BIZ-01, BS-BIZ-02 | **核心业务流**：自动操作，但手动验证弹窗细节和状态显示更可靠 |
| 权限校验 | **🔄 混合** | PERM-01, PERM-02 | 前端界面自动点一点，后端接口调用建议用自动化接口测试 |
| 异常场景 (网络/超时) | **❌ 不建议自动化** | EXC-01, EXC-02 | 手动验证用户体验反馈（UI 提示文字、按钮状态） |
| 并发冲突 / 数据一致性 | **❌ 不建议自动化** | BS-BIZ-05 | 需要两个真实 session/connection，UI 层模拟复杂，性价比低。建议用接口测试覆盖 |
| 驳回意见校验 | **✅ 适合自动化** | BS-BIZ-03 | 前端逻辑明确，可自动验证按钮状态或接口拦截 |
| 空数据与无数据状态 | **✅ 适合自动化** | UI-02 | 可用 mock 数据提前准备空数据环境 |

> **总结**: 约 60% 的测试场景适合完全自动化，30% 适合混合（自动+手动检查），10%（并发、网络异常等）不建议自动化，更适合手动或接口测试。

## 不推荐自动化的场景

| 场景 | 原因 |
|------|------|
| 网络中断 (TC-EA-018) | 依赖系统网络状态，UI层模拟难度高，用户体验反馈（toast等）手动观察更准确 |
| Token 过期 (TC-EA-020) | 涉及登录态管理，手动构造过期Token更可控，UI自动模拟复杂 |
| 并发冲突 (BS-BIZ-05) | 需要两个真实浏览器session，UI层并发执行逻辑复杂，建议用接口测试覆盖 |
