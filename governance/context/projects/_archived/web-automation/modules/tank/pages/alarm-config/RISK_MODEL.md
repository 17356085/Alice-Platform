# 业务场景模型: tank/alarm-config

## 1. 业务目标
- **核心业务目标**：为每个储罐配置合理的报警阈值和行为，实现对储罐异常状态的早期预警和精确记录，确保生产和运营安全。
- **次要业务目标**：提供报警规则的统一管理、启用/禁用、同步和审计功能，方便运维人员批量维护。
- **非目标**：本页面**不**负责处理报警产生后的推送、分发和执行（那是`alarm-execution`模块的职责）；也不负责生成报警统计报表。

## 2. 角色与旅程

| 角色 | 职责边界 | 典型旅程 |
|---|---|---|
| **运营管理员 (admin)** | 创建、修改、删除任何储罐的报警规则，操作不受限制。 | 1. 登录系统。<br>2. 通过导航进入 “储罐监控 -> 报警配置”。<br>3. 在列表页通过“储罐名称”或“区域”筛选器定位到目标储罐（如T-101）。<br>4. 点击“编辑”，在弹窗中修改“液位高点报警”的阈值为“10.5m”。<br>5. 点击“保存”，确认后端返回成功信息。 |
| **安全工程师 (safety_eng)** | 仅能编辑和查看其负责区域的储罐报警规则，无权删除或全局禁用。 | 1. 登录系统，看到已被权限过滤后的储罐列表（仅包含区域A）。<br>2. 发现T-101的“温度”报警触发了频繁报警。<br>3. 点击“编辑”，将报警延迟从“0秒”修改为“30秒”，减少误报。<br>4. 查看历史操作日志，确认修改记录。 |
| **值班操作员 (operator)** | 只能查看报警规则列表和当前状态，无任何修改、删除、新增权限。 | 1. 登录系统，查看全厂/本区域的报警规则概览。<br>2. 在列表页，看到T-201的“高液位报警”状态为“已禁用”。<br>3. 向安全工程师汇报“T-201的高液位报警被禁用了”。 |
| **审计员 (auditor)** | 仅可查看操作日志，无直接页面访问权限。 | 1. 从审计报告链接进入“操作日志”页面。<br>2. 筛选模块“报警配置”，查看谁在何时修改了任何报警规则。 |

## 3. 业务流程

本页面参与的主要流程：

- **主流程 (Happy Path)**:
    > 监控中心 -> 新建报警规则(或从模板导入) -> 填写/配置常规报警(液位、温度、压力、BOG) -> 设置阈值和级别(P0/P1/P2) -> 保存 -> 列表页刷新显示新规则 (流程结束于 `tank/alarm-config`)

- **分支流程1: 批量禁用关键报警**
    > 列表页 -> 勾选多个储罐 -> 点击“禁用”按钮 -> 弹窗二次确认 -> 确认 -> 列表更新状态为“禁用” (流程在页面内完成)

- **异常流程: 保存时接口超时**
    > 编辑弹窗 -> 点击“保存” -> 按钮进入Loading状态 -> 等待5秒后接口超时 -> 前端弹出Toast信息：“保存请求已发送，请稍后刷新页面确认状态。” -> 用户手动刷新页面 -> 查看规则状态是否已更新 -> 若未更新，重新进行编辑保存流程。

## 5. 数据流

- **数据来源**：
    - `tank/summary` (储罐基础信息页): 创建报警规则时，储罐名称、编号、区域、设计容量等信息**自动带入**，不可修改。
    - `system/template` (报警模板页 - 推测): 用户可以选择“导入模板”功能，快速创建一组标准报警规则。
- **数据变换**：
    - 本页面执行 **CRUD** 操作。用户输入的业务规则（阈值、级别、延迟等）被转换为结构化的JSON数据，通过API提交到后端。
- **数据消费**：
    - `alarm/processing` (报警处理模块): 实时读取本页面配置的报警规则，用于判断实时采集的现场数据（液位、温度等）是否触发报警。
    - `report/analysis` (报表分析): 读取配置历史，分析报警设置的有效性和合理性。
