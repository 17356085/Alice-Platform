## 逐项详细分析

### 建议 1: 拆分长会话

**方案**: 50+ turn 的对话拆为多次短会话

**风险评估**: 🟢 低

| 风险点 | 分析 | 缓解 |
|--------|------|------|
| 上下文丢失 | 新会话需重新加载 CLAUDE.md + Memory，AI 需重新理解当前状态 | 会话结束时执行 `context-sync` 写 CURRENT_TASK 标记进度；下次用 "上次做到哪了" 触发恢复 |
| 重复解释 | 用户可能需要重新描述问题 | 短会话聚焦单任务，反而减少无关上下文污染 |
| workflow 中断 | /full-sop 的断点续跑依赖 SOP_STATUS.json 和 SqliteSaver checkpoint | 不受影响——这两个都在文件系统中 |

**结论**: 无系统风险。唯一成本是用户习惯调整。

---

### 建议 2: 精简 CLAUDE.md 路由表

**方案**: 12 条口语化路由 → 3 条高频路由 + 外部引用

**风险评估**: 🟡 中

| 风险点 | 严重度 | 详细分析 |
|--------|--------|---------|
| **误路由** | 🔴 高 | 删除"测XX模块"→`/full-sop` 的强制规则后，AI 可能直接跳到 `/automation-agent` 绕过编排器。CLAUDE.md 第 41 行的 `⛔ 强制路由` 是 SOP 合规的关键防线 |
| 新增 Agent 遗漏 | 🟡 中 | 如果未来新增 Agent，外部路由文件可能被遗忘更新 |
| 新 AI 首次加载 | 🟢 低 | 新 AI 读 CLAUDE.md 时可能看不到完整路由，但可以通过引用找到 |

**受影响的关键路由**:
```
⛔ "测XX"/"自动化XX"/"给XX写测试" → /full-sop  ← 这条绝对不能丢
✅ "这个按钮怎么写定位" → /automation-agent      ← 可以外移
✅ "这段代码写得对不对" → code-consistency-checker ← 可以外移
```

**缓解措施**: 保留 3 条强制路由 + SOP 核心约束 + 代码红线在 CLAUDE.md，其余口语化路由移到 `governance/docs/reference/routing-guide.md` 并在 CLAUDE.md 中加一句 `> 完整路由见 governance/docs/reference/routing-guide.md`

**结论**: 可以执行，但**必须保留**第 41 行的强制路由规则和第 99-110 行的代码红线。

### 建议 4: Skill 门禁去重

**方案**: 8 个 SKILL.md 中重复的 SOP 门禁说明提取为共享引用

**风险评估**: 🔴 高 — **不推荐执行**

| 风险点 | 严重度 | 详细分析 |
|--------|--------|---------|
| **共享引用加载失败** | 🔴 致命 | Claude Code 的 Skill 机制通过 `source:` 和文件内容工作。如果门禁逻辑移到外部文件，而该文件在 Skill 执行时未被加载，SOP 门禁被静默跳过 |
| **8 个 Agent 全受影响** | 🔴 致命 | 门禁是 L2 防线。任一 Agent 跳过门禁都会允许缺失上游产物的情况下执行，违反 SOP 核心约束 |
| **回滚困难** | 🟡 中 | 发现问题时 8 个文件已被修改 |
| 独立性丧失 | 🟡 中 | Skill 文件当前自包含——直接读 `SKILL.md` 就获得全部指令。引入外部引用后，理解单个 Skill 需要跨文件追踪 |

**为什么门禁重复在这里是合理冗余**:
```
每个 SKILL.md 的门禁部分 (~15 行):
  1. 提取 module 参数
  2. 运行 check_sop_gate.py --module <m> --agent <name> --json
  3. 检查 blocked 字段
  4. blocked=true → 拒绝并输出阻断信息

这 15 行对每个 Agent 基本相同但 run_sop_gate.py 的参数不同 (--agent 参数)
```
真正完全相同的只是步骤 2-3 的模式，但 `--agent` 参数和阻断信息每 Agent 不同。去重的实际节省远小于估计——可能只有 ~50 行共享，而不是 ~200。

**结论**: ❌ 不推荐。节省 vs 风险不成比例。L2 防线不能有单点故障。

### 建议 5: 巨型 Skill 拆分

**方案**: `excel-exporter.md` (22KB) 和 `knowledge-manager.md` (9.4KB) 拆分为指令 + 数据

**风险评估**: 🔴 高 — **不推荐**

| 风险点 | 严重度 | 详细分析 |
|--------|--------|---------|
| **excel-exporter 功能断裂** | 🔴 致命 | 22KB 中的大量示例表格定义了 Excel 输出格式。拆分后如果示例不被加载，生成的 Excel 可能格式错误、缺少列、数据错位 |
| **knowledge-manager 知识提取逻辑损伤** | 🔴 高 | 9.4KB 包含知识沉淀的完整规则链条。精简可能丢失边界条件判断 |
| 测试覆盖不足 | 🟡 中 | 这两个 Skill 没有独立的单元测试，拆分后的功能正确性完全依赖人工验证 |
| 下游 Agent 依赖 | 🟡 中 | `report-agent` 依赖 excel-exporter 的输出格式。格式变更可能导致 `report-agent` 读错数据 |

**excel-exporter.md (22KB) 内容分析**:
- 约 15KB 是示例表格和输出格式模板
- 约 3KB 是指令
- 约 4KB 是边界条件/错误处理

这些示例表格是 Skill 的核心——没有它们，LLM 生成的 Excel 格式不可控。不能简单截断。

**结论**: ❌ 不推荐。风险远大于收益。如果确实需要优化，应该：
1. 先为这两个 Skill 写测试
2. 用测试验证拆分不会破坏输出
3. 逐步迁移
