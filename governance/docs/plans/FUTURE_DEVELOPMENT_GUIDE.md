## 二、Context 版本化方案

### 触发条件

当你开始以下任一场景时执行：
- 被测系统有多个并行版本（如 v2.x 和 v3.x 同时维护）
- 多人同时修改不同模块的 PAGE_CONTEXT
- 出现"测试脚本与页面实际结构对不上"的问题且难以追溯

### 推荐方案

```
方案: 轻量 Git 驱动版本化（非数据库方案）

1. PAGE_CONTEXT.md 头部添加元数据块:
   ```yaml
   ---
   version: 2
   last_verified: 2026-06-13
   tested_commit: a1b2c3d
   tested_by: automation-agent
   ---
   ```

2. consistency_checker.py 新增检查:
   - PAGE_INTERFACE.yaml 的 tested_commit 是否在 PAGE_CONTEXT.md 之后
   - 如果代码仓库有新 commit 但 PAGE_CONTEXT 超过 30 天未验证 → 警告

3. 不做的事:
   - 不要建数据库存储版本历史 → Git history 本身就是版本历史
   - 不要做自动回滚 → 风险太高，人工确认更安全
```

### 预估工作量: 2-3h

## 三、第三类方向决策理由

以下方向在架构评审中被标记为"不值得做"。此处记录决策理由，供后续参考。

### 3.1 Event Bus 实时化（Kafka/Redis）

**为什么不做**: 当前 Event Bus 的唯一消费者是 Knowledge Agent，它在测试周期结束时运行。60s 轮询延迟对于这个场景完全可接受。引入消息队列会：
- 增加部署依赖（需要运行 Kafka/Redis）
- 增加维护负担（消息丢失、重试、死信队列）
- 不带来实际价值（消费者不需要实时性）

**什么时候重新考虑**: 当出现 3 个以上异构消费者（如 Slack 通知、CI 触发器、Dashboard 实时推送）时。

### 3.2 Task Queue 持久化

**为什么不做**: 当前任务队列的使用模式是"触发即执行"，任务生命周期 < 5 分钟。SQLite 持久化（task_queue.py 已实现）在重启场景下可以恢复，但：
- 当前没有长时间运行的任务
- 重启后恢复一个"正在生成 PageObject"的任务没有意义（LLM 调用已中断）
- 用户更倾向于重新触发而非恢复

**什么时候重新考虑**: 当出现耗时 > 10 分钟的任务或批量调度场景时。

### 3.3 真正的 Multi-Agent 协作

**为什么不做**: 当前 8 个 Agent 是串行流水线（Phase 0→1→2→...），这是测试自动化的自然流程。引入并行 Agent 协作需要：
- 重写 `AgentLoop.run()` 核心循环
- 引入共享工作内存（当前通过文件系统传递）
- 处理 Agent 间冲突（两个 Agent 同时修改同一个 PageObject？）
- 投入至少 2-4 周

而当前的串行流水线覆盖了所有必要的工作，且产出质量可控。

**什么时候重新考虑**: 当需要在同一个模块上并行执行"测试设计"和"自动化"时（目前它们是串行依赖关系，无法并行）。

### 3.4 A2A 协议

**为什么不做**: A2A（Agent-to-Agent）协议是 Google 提出的跨平台 Agent 通信标准。当前：
- 只有一个项目（Web 自动化）
- 没有外部 Agent 需要与本平台交互
- MCP 协议已经提供了标准化的外部接口

**什么时候重新考虑**: 当出现第二个独立的 AI 测试项目（如移动端自动化），且需要与 Web 端 Agent 共享知识库时。

### 3.5 Web Dashboard / 可视化平台

**为什么不做**: `aitest dashboard` CLI 和 `/health` API 端点已提供足够的可观测性。Web Dashboard 需要：
- 前端框架选择和维护
- 用户认证系统
- 持续的 UI 迭代

对于单人项目，CLI + API 的组合是最高效的交互方式。

**什么时候重新考虑**: 当团队规模 ≥ 3 人，或需要给非技术人员展示进度时。

### 3.6 Dify 集成

**为什么不做**: Dify 是 LLM 应用编排平台。引入它会：
- 创造与 LangGraph 的功能重叠
- 增加外部平台依赖
- 将 Skill Prompt 管理迁移到 Dify 的格式

当前 LangGraph + MCP Server 已经提供了完整的编排和对外接口。

**什么时候重新考虑**: 当 LangGraph 维护成本超过其价值，且团队已有 Dify 使用经验时。

## 四、下一次架构跃迁的触发条件

当前架构评级 B+/A-，对单人项目是健康的。以下任一条件触发时，应进行下一次架构复审：

| 触发条件 | 需要重新思考的部分 |
|---------|------------------|
| 第二个项目接入（如移动端自动化） | 多项目 Context 管理、知识库隔离、Agent 复用 |
| 第二个人加入维护 | 文档完整性、测试覆盖率、开发环境标准化 |
| Skill 数量超过 30 | Skill 参数化机制、Prompt 版本管理工具 |
| 被测系统技术栈变更（如 Vue 3 → React） | Element Plus 特化定位器 → 通用化 |
| LLM API 范式变化（如 MCP 成为模型内置能力） | MCP Server 层的必要性 |
| LangGraph 发布 1.0（当前 0.x） | 迁移评估、breaking changes 影响分析 |

## 六、优先级速查

| 做什么 | 什么时候做 |
|--------|----------|
| 继续用平台产出测试用例 | **现在** — 这是平台存在的目的 |
| 修复实际使用中遇到的 bug | 随时 |
| 添加新模块的 PAGE_CONTEXT | 按需 |
| Context 版本化 | 多人协作或多版本并行时 |
| 下一个架构复审 | 触发条件满足时（见第四章） |
| Event Bus 实时化 | 有 3+ 异构消费者时 |
| Multi-Agent | 有并行执行需求时 |
| Web Dashboard | 团队 ≥ 3 人时 |
