# Skill: testcase-design

## 目标
从模块上下文和页面上下文出发，输出高质量测试设计与测试用例草案。

## 输入
- MODULE_CONTEXT
- PAGE_CONTEXT
- 需求说明
- 风险信息
- 历史缺陷

## 输出
- TEST_DESIGN.md
- TEST_CASES.md
- 自动化优先级建议

## 规则
- 先场景，后步骤
- 先风险，后覆盖
- 区分手工、自动化、混合
- 明确不建议自动化的场景

## 依赖
- templates/test-design.template.md
- templates/test-cases.template.md

---

## Prompt 模板

> 以下 Prompt 可直接复制到 AI 对话中使用。替换 `{{ }}` 占位符即可。

### 页面级测试设计

```text
基于以下页面上下文和风险模型，设计完整的测试方案。

## 输入
- PAGE_CONTEXT：{{粘贴核心内容}}
- RISK_MODEL：{{粘贴核心内容}}

## 任务
输出 TEST_DESIGN.md，按以下维度设计测试：

### 1. 页面加载与显示
- 正常加载、空数据状态、权限受限状态
- 各区域元素完整性校验

### 2. 搜索与筛选
- 单条件搜索、组合搜索、模糊搜索
- 重置筛选、切换筛选条件
- 特殊字符/超长文本搜索

### 3. 表格操作
- 排序、列宽调整、数据格式化
- 行选中、批量操作
- 分页：首页/末页/跳转/改变每页条数

### 4. 新增操作
- 打开弹窗、表单校验（必填项/格式/边界值）
- 提交成功/失败、取消操作
- 重复提交、网络异常时的处理

### 5. 编辑操作
- 数据回显完整性
- 修改后保存、取消修改
- 并发编辑冲突

### 6. 删除操作
- 单个删除、批量删除
- 删除确认弹窗、取消删除
- 删除关联数据时的处理

### 7. 权限测试
- 不同角色看到的功能差异
- 无权限操作的拦截

### 8. 异常场景
- 网络断开、接口超时、500错误
- Token过期、会话超时

## 约束
- 每个测试点格式：| 编号 | 测试场景 | 优先级 | 前置条件 | 测试步骤 | 预期结果 |
- P0（阻塞上线）、P1（影响核心体验）、P2（边缘场景）
- 与 RISK_MODEL 中的风险项一一对应
```

### 生成详细测试用例表

```text
基于以下测试设计方案，生成可执行的详细测试用例表。

## 输入
TEST_DESIGN：{{粘贴 TEST_DESIGN.md}}

## 任务
将每个测试场景展开为可执行用例，输出 TEST_CASES.md：

| 字段 | 说明 |
|------|------|
| 用例编号 | TC-{模块缩写}-{序号}，如 TC-USER-001 |
| 用例标题 | 简洁描述测试目标 |
| 所属模块 | {{模块名}} |
| 所属页面 | {{页面名}} |
| 优先级 | P0/P1/P2 |
| 前置条件 | 数据准备、账号、环境要求 |
| 测试步骤 | 编号的详细操作步骤（可被另一个人复现） |
| 测试数据 | 具体的输入值（不要写"输入用户名"，写"输入 admin"） |
| 预期结果 | 精确的预期行为描述 |
| 实际结果 | （留空，执行时填写） |
| 自动化 | ✅ 已自动化 / 🔄 待开发 / ❌ 不适合 |

## 格式要求
- 用例编号连续
- P0 用例 100% 覆盖，P1 ≥ 80%，P2 ≥ 50%
```

## 检查清单

### TEST_DESIGN 检查
- [ ] 覆盖 8 个测试维度（加载/搜索/表格/新增/编辑/删除/权限/异常）
- [ ] 每个测试点有唯一编号
- [ ] 每个测试点有优先级（P0/P1/P2）
- [ ] 每个测试点有前置条件和预期结果
- [ ] P0 场景与 RISK_MODEL 中的 P0 风险一一对应
- [ ] 明确标注哪些场景适合手工、哪些适合自动化
- [ ] 明确标注不建议自动化的场景及原因

### TEST_CASES 检查
- [ ] 用例编号格式正确、连续不重复
- [ ] 测试数据具体可执行（不是"输入用户名"而是"输入 admin"）
- [ ] P0 覆盖率 100%
- [ ] P1 覆盖率 ≥ 80%
- [ ] 每个用例标注自动化状态
- [ ] 与已有用例格式一致
