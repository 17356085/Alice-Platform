# Skill: risk-modeling

## 目标
从页面上下文出发，识别业务风险、权限风险、数据风险、接口风险等，输出页面级风险模型。

## 输入
- PAGE_CONTEXT.md
- MODULE_CONTEXT.md
- 历史缺陷记录
- 权限矩阵（如有）

## 输出
- RISK_MODEL.md
- BUSINESS_SCENARIOS.md（P2-5: 业务场景模型 — testcase-design 的必选输入）

## 规则
- 从 6 个维度识别风险：业务、权限、数据、接口、UI/UX、性能
- 风险等级：P0（阻塞上线）/ P1（影响核心体验）/ P2（边缘场景）
- 每个 P0 风险必须给出缓解措施
- 关联现有自动化用例覆盖率，标注未覆盖的高风险场景
- **P2-5: 风险建模后必须产出 BUSINESS_SCENARIOS.md**，从业务目标→角色→流程→规则→数据的层次描述业务场景

## 依赖
- templates/risk-model.template.md
- skills/page-analysis.md
- skills/module-modeling.md（获取模块业务上下文）

## 边界
- 本 Skill 不产出测试设计（那是 testcase-design 的职责）
- 本 Skill 不产出自动化策略（那是 auto-strategy 的职责）
- BUSINESS_SCENARIOS.md 描述"业务上要验证什么"，不描述"UI 上怎么操作"

### 页面级风险建模

```text
基于以下页面上下文，进行全面的风险建模。

## 页面信息
- 页面：{{用户列表页}}
- 模块：{{系统管理-用户管理}}
- PAGE_CONTEXT：{{粘贴 PAGE_CONTEXT.md 核心内容}}

## 任务
从以下维度识别风险，输出 RISK_MODEL.md：

### 1. 业务风险
- 误操作导致的数据丢失（如：误删用户）
- 权限越权（如：普通用户看到管理功能）
- 业务规则冲突

### 2. 权限风险
- 不同角色看到不该看的数据
- 无权限用户通过URL直接访问
- 接口未校验权限

### 3. 数据风险
- 数据边界（空值、超长、特殊字符、SQL注入）
- 数据一致性（删除关联数据时的级联处理）
- 并发操作（两人同时编辑同一记录）

### 4. 接口风险
- 接口超时/失败时的前端表现
- 接口返回异常数据格式
- Token过期处理

### 5. UI/UX风险
- 不同分辨率下的布局问题
- 加载状态的用户体验
- 错误提示是否友好

### 6. 性能风险
- 大数据量下的表格渲染
- 频繁搜索时的请求频率

## 输出要求
- 每个风险标注等级（P0/P1/P2）
- P0 = 阻塞上线，P1 = 影响核心体验，P2 = 边缘场景
- 每个P0风险必须给出缓解措施
- 关联现有自动化用例覆盖率，标注未覆盖的高风险场景
```

### 业务场景建模（P2-5: 产出 BUSINESS_SCENARIOS.md）

```text
基于以下页面上下文和风险模型，分析业务场景。目标：从业务出发理解测试需求，而非从 UI 操作出发。

## 输入
- PAGE_CONTEXT：{{粘贴 PAGE_CONTEXT.md 核心内容}}
- RISK_MODEL：{{粘贴 RISK_MODEL.md 核心内容}}
- MODULE_CONTEXT：{{粘贴 MODULE_CONTEXT.md 中的业务描述}}

## 任务
输出 BUSINESS_SCENARIOS.md，从以下 6 个维度分析业务场景：

### 1. 业务目标 (Business Goal)
本页面服务于什么业务目标？用户在真实业务中为什么进入此页面？
- 核心业务目标（1-2 句话）
- 次要业务目标
- 非目标（明确本页面不解决什么）

### 2. 角色与旅程 (Roles & Journeys)
哪些角色使用此页面？各自完成什么任务？
- 角色列表（角色名 + 职责边界）
- 每个角色的典型业务旅程（从进入到离开的完整过程）
- 多角色协作场景（如有）

### 3. 业务流程 (Workflows)
本页面参与哪些端到端业务流程？
- 跨页面主流程（标注本页面在流程中的位置）
- 分支流程
- 异常中断及恢复流程
- 流程描述（至少覆盖 Happy Path + 2 条 Alternative Path）

### 4. 业务规则 (Business Rules)
本页面涉及哪些业务规则（非 UI 校验规则）？
- 业务校验规则（如：液位不能超过罐容的 95%）
- 状态流转规则（如：待审批→已审批→已执行）
- 计算规则（如：BOG = f(温度, 压力, 液位)）
- 触发规则（如：温度超阈值 → 自动生成报警工单）

### 5. 数据流 (Data Flow)
数据从哪来、经过什么变换、到哪去？
- 数据来源（本页面数据从哪个页面/接口创建）
- 数据变换（本页面对数据做了什么操作）
- 数据消费（本页面的数据被哪些下游页面/模块使用）
- 跨页面数据一致性要求

### 6. 业务风险→场景映射 (Risk-to-Scenario Mapping)
- 将 RISK_MODEL 中的每个 P0/P1 风险映射到具体业务场景
- 标注哪些业务场景对应哪些风险
- 标注当前缺失覆盖的业务场景

## 输出要求
- 每个维度至少 3 条具体内容（非空泛描述）
- 业务规则必须是可验证的具体规则（不是"校验输入合法性"这种泛化描述）
- 数据流必须标注具体的页面/模块名称
- 角色旅程必须可操作（另一个测试人员能照着执行）
- 标注信息置信度：✅ 已确认 / ⚠️ 推断 / ❓ 待确认

## 格式
| 场景ID | 业务维度 | 场景描述 | 涉及角色 | 涉及页面 | 关联风险 | 置信度 |
|--------|----------|----------|----------|----------|----------|--------|
| BS-XXX-001 | 业务目标 | ... | admin | page-a, page-b | RISK-001 | ✅ |
```
