# Skill: bug-analysis

## 目标
将失败现象转化为标准化 Bug 分析结论。支持 RAG 自动匹配已知问题。

## 输入
- 失败日志
- 截图
- 复现步骤
- 代码上下文
- CI 结果

## 输出
- Bug 分析报告
- 根因分类
- 修复建议
- 回归建议

## 🔴 MANDATORY: 执行流程（不可跳过）

### L0：自动 RAG 已知问题匹配（🔴 必须执行，不可跳过）

在分析任何失败用例之前，**必须先调用 RAG 检索已知问题库**：

1. **提取错误签名**（对每个失败的测试用例）：
   - 异常类型（如 `NoSuchElementException`, `TimeoutException`, `AssertionError`, `ElementClickInterceptedException`）
   - 关键报错信息（如 “el-cascader”, “loading”, “遮罩”, “teleport”）
   - 失败的页面/模块名称
   - 涉及的 Element Plus 组件名

2. **调用 MCP Tool `rag_search_known_issues`**：
   ```json
   {
     “query”: “<异常类型> <报错关键信息> <涉及组件名> <页面名称>”,
     “n_results”: 5
   }
   ```

3. **对返回结果分类**：
   - 若 `distance < 0.5`（极高相似度）→ 🟢 **直接匹配**，引用已知坑位的解决方案，跳过 L1 深度分析
   - 若 `distance < 1.0`（高相似度）→ 🟡 **可能已知问题**，参考已知坑位作为分析方向
   - 若 `distance >= 1.0` 或无结果 → 🔵 未匹配，正常进入 L1 深度分析

4. **将匹配结果写入 BUG_ANALYSIS 报告的「已知问题匹配」栏**：
   ```markdown
   ## 已知问题匹配
   | 匹配编号 | 标题 | 相似度 | 处理方式 |
   |---------|------|--------|---------|
   | EP-001 | Teleport 渲染 | 0.32 | 🟢 直接匹配，引用解决方案 |
   | EP-003 | loading 遮罩 | 0.87 | 🟡 参考方向 |
   ```

### 降级规则

- 如果 MCP Tool `rag_search_known_issues` 调用失败（超时/服务不可用）：
  → 降级为手动读取 `governance/context/known-issues.yaml`
  → 基于关键字做简单匹配（在 YAML 中搜索异常类型/组件名/症状关键词）
  → 在报告中标注「⚠️ RAG 不可用，使用降级匹配」
- 如果 known-issues.yaml 也不可读：
  → 跳过已知问题匹配，直接进入深度分析
  → 在报告中标注「⚠️ 已知问题匹配不可用」

### L1.0：构建反馈循环（🔴 在深度分析前执行）

> 方法论来源: [mattpocock/skills — /diagnose](https://github.com/mattpocock/skills)

**核心原则**: 如果你有一个快速、确定性、Agent 可运行的 pass/fail 信号，bug 90% 已经修复。没有反馈循环，盯着代码看是浪费时间。

#### 构建方式（按推荐顺序尝试）

1. **Failing test** — 在离 bug 最近的可测试接缝处写一个失败的 pytest 用例。首选。
2. **curl / HTTP 脚本** — 针对 API 级 bug，直接构造请求比对响应。
3. **CLI 调用 + fixture diff** — 固定输入，比对 stdout 与已知正确快照。
4. **Playwright / BrowserUse 脚本** — 操作 UI，断言 DOM / Console / 网络请求。适合 Element Plus Teleport 类 bug。
5. **回放捕获的 trace** — 把真实的网络请求 / payload / event log 存盘，隔离回放。
6. **最小化 throwaway harness** — 只启动相关服务子集 + mock 依赖，单函数入口。
7. **Property / fuzz loop** — 若 bug 是"偶尔输出错误"，跑 1000 次随机输入找失败模式。
8. **Bisection harness** — 若 bug 出现在两个已知状态之间（commit / 数据集），自动化 `git bisect run`。
9. **Differential loop** — 同一输入跑旧版本 vs 新版本，diff 输出。
10. **HITL bash 脚本** — 最后手段。若必须人工点击，用 `scripts/hitl-loop.template.sh` 结构化人工流程。

#### 迭代反馈循环自身

有了第一个 loop 后问：

- 能更快吗？（缓存 setup，跳过无关 init，缩小测试范围）
- 信号能更尖锐吗？（断言到具体症状，不只是"没崩"）
- 能更确定吗？（固定时间/seed/RNG，隔离文件系统，冻结网络）

**30 秒的 flaky loop ≈ 没有 loop。2 秒的确定性 loop = 调试超能力。**

#### 非确定性 bug

目标不是完美复现，而是**提高复现率**。loop 跑 100 次并行，加 stress，缩小时间窗口，注入 sleep。50% flaky → 可调试；1% flaky → 不可调试 → 继续提高复现率。

#### 确实无法构建 loop 时

停止，明确说明。列出尝试过的方式。向用户请求：(a) 能复现的环境访问权限，(b) 捕获的 artifact（HAR / log dump / 录屏+时间戳），(c) 临时生产环境插桩授权。**没有 loop 不要进入 L1。**

### L1：深度分析（仅在 RAG 未直接匹配且反馈循环已建立时执行）
执行标准 5 层递进排查：定位器失效 → 等待不足 → 数据问题 → 环境问题 → 产品Bug

## 规则
- **L0 (RAG 自动匹配)**：失败后优先检索 known_issues 向量库 → 匹配已知坑位 → 直接引用解决方案
- 区分现象、证据、根因、建议
- 不能确认的结论必须标记”待验证”
- 必须输出影响范围与回归范围
- L0 步骤在任何情况下都不可跳过（即使 AI 认为”已知问题不相关”也必须执行检索）
