# Skill: project-context-manager

## 目标
识别测试项目、模块上下文、事实源和迁移边界。

## 输入
- 项目结构
- 项目级上下文
- 模块上下文
- SOP / Prompt / README / Jenkins

## 输出
- 项目索引
- 模块索引
- 事实源映射
- 缺失上下文清单

## 规则
- 优先引用，不急于复制旧资产
- 模块是治理核心单元
- 项目级只保留稳定共性
- 页级细节不塞入项目总览文件

## 依赖
- context/project-index.yaml
- context/source-of-truth.md

## Prompt 模板

```text
你是一个高级软件测试架构师。你需要为一个新项目建立完整的项目级测试上下文文档。

## 项目信息
- 项目名称：{{鞍集涂源管理系统}}
- Web端地址：{{http://8.136.215.171:8081/}}
- 前端技术栈：{{Vue 3 + Element Plus}}
- 测试账号：{{admin / 密码}}
- 平台类型：Web管理端 + 微信小程序端

## 任务
请输出 PROJECT_CONTEXT.md，包含以下章节：

1. **项目概述**：系统定位、业务领域、用户群体
2. **技术架构**：前端/后端/数据库/部署方式
3. **用户角色与权限模型**：列出所有角色及其权限边界
4. **功能模块树**：一级菜单→二级菜单→页面，完整层级关系
5. **核心业务流程**：最重要的3-5条业务链路
6. **测试风险矩阵**：按模块评估测试风险（高/中/低）
7. **自动化优先级建议**：P0（必须自动化）/ P1（建议）/ P2（手工）
8. **已完成的测试工作**：列出已覆盖的模块及用例数
9. **踩坑经验**：已知的技术难点和解决方案

## 约束
- 使用中文
- 表格优于纯文字列表
- 模块树要完整，不要遗漏
- 每个风险项给出缓解措施
```

2. 仅输出变更部分（diff 格式或"替换段落"格式）
3. 不修改未被此变更影响的内容
4. 新坑位需分配编号（EP-XXX，延续现有编号）
5. 更新后检查事实源一致性
