# Skill: requirement-analysis

## 目标
从 PRD、原型图或产品说明中识别需求风险，输出测试视角的需求分析文档。

## 输入
- PRD / 原型图 / 产品说明
- 所属模块和项目背景
- 已有 MODULE_CONTEXT（如有）

## 输出
- 需求理解摘要
- 业务规则清单（显式 + 隐式）
- 测试范围界定（纳入 / 不纳入）
- 需求风险点（模糊、矛盾、遗漏）
- 测试策略建议（手工/自动化/混合）
- 测试计划与迭代建议
- 自动化机会清单

## 规则
- 先理解需求，再分析测试
- 明确标注"不纳入测试"的范围及原因
- 每项风险必须给出缓解或应对措施
- 不确定的规则标注"待产品确认"

## 依赖
- workflows/sop-summary.md (§ Phase 0.8)

## 边界
- 本 Skill 不产出模块上下文（那是 module-modeling 的职责）
- 本 Skill 不产出详细测试设计（那是 testcase-design 的职责）

### 需求分析与测试计划

```text
你是一个测试分析师。请对以下需求进行测试视角的分析。

## 需求材料
{{粘贴 PRD 内容或原型图描述}}

## 项目背景
- 项目：{{鞍集涂源管理系统}}
- 模块：{{人员管理-培训管理}}
- 平台：{{Web管理端 / 微信小程序端}}

## 任务
输出需求分析文档，包含：

1. **需求理解**：用一句话概括需求目标
2. **业务规则提取**：列出所有显式和隐式的业务规则
3. **测试范围界定**
   - ✅ 纳入测试：{{明确要测的功能}}
   - ❌ 不纳入测试：{{明确不测的功能及原因}}
4. **风险识别**：
   - 需求模糊点
   - 逻辑矛盾/遗漏
   - 与技术架构的冲突
5. **测试策略建议**：手工/自动化/混合？原因？
6. **测试计划**：分几个迭代？每个迭代测什么？
7. **自动化机会**：哪些功能点适合自动化？优先级？

## 约束
- 不确定的规则标注"待产品确认"
- 每项风险必须给出缓解或应对措施
```

## 示例

### 输入
```
需求：设备管理模块新增"批量导入设备"功能
PRD：上传 Excel → 校验 → 预览 → 确认导入 / 下载失败记录
平台：Web 管理端
```

### 输出（节选）
```
1. 需求理解：允许用户通过 Excel 模板批量导入设备信息，替代逐条新增。

2. 业务规则：
   显式：Excel 模板固定列（名称/类型/编号/位置），必填列不可为空
   隐式：设备编号需唯一，重复时跳过并记录

3. 测试范围：
   ✅ 纳入：模板下载、校验规则、边界数据、失败重试
   ❌ 不纳入：Excel 文件内容解析引擎（第三方库，已有单元测试）

4. 风险识别：
   - 模糊点：10000+ 条记录时是否异步处理？（待产品确认）
   - 遗漏：未定义导入失败时的回滚策略
   
5. 测试策略：混合。核心校验逻辑自动化，UI交互手工验证。

6. 测试计划：
   迭代1：模板下载 + 单条/少量数据导入
   迭代2：边界数据 + 异常模板 + 大数据量
   
7. 自动化机会：
   - P0：模板下载接口、校验规则（参数化测试）
   - P1：导入结果验证
```
