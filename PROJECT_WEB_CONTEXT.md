## 2. 技术架构

### 2.1 前端

| 层级 | 技术 | 版本/说明 |
|------|------|----------|
| 框架 | Vue 3 | Composition API + `<script setup>` |
| UI 组件库 | Element Plus | el-table / el-dialog / el-form / el-pagination / el-tree / el-select / el-radio-group |
| 构建工具 | Vite | 按路由懒加载，代码分割 |
| 状态管理 | Pinia | 持久化至 localStorage |
| 路由 | Vue Router 4 | **Hash 模式** (`#/system/user`)，动态路由从后端加载 |
| HTTP 客户端 | Axios | 请求/响应拦截器，自动 refreshToken |
| 图标 | Element Plus Icons + Remix Icons + Tabler Icons | 多图标集混用 |
| CSS | Scoped CSS | 组件样式隔离 |

### 2.2 后端

| 层级 | 技术 | 说明 |
|------|------|------|
| 语言/框架 | Java + Spring Boot | RESTful API |
| API 前缀 | `/api/` | 统一 JSON 响应 `{ code, message, data, timestamp }` |
| 鉴权 | JWT | accessToken 2h + refreshToken 30d，双 token 自动续期 |
| 权限模型 | RBAC | 角色 → 菜单权限 + 按钮权限 + 数据范围 (DataScope) |
| API 文档 | 接口管理模块内置 | `/api` 接口在线调试 |

### 2.3 数据库

| 层级 | 技术 | 说明 |
|------|------|------|
| 数据库 | MySQL | 关系型数据存储 |
| 文件存储 | MinIO / OSS | 文件上传/代理下载 |

### 2.4 部署方式

| 层级 | 方式 | 说明 |
|------|------|------|
| 前端 | Nginx 静态部署 | Vite 构建产物 |
| 后端 | Spring Boot Jar | 独立部署或容器化 |

## 3. 用户角色与权限模型

### 3.1 角色定义

| 角色标识 | 角色名称 | 权限边界 | 适用场景 |
|----------|----------|----------|----------|
| `admin` | 超级管理员 | **全部功能**，`isAdmin=true` 跳过所有权限检查；DataScope=1（全部数据） | 系统运维 |
| `commonAdmin` | 普通管理员 | 系统管理 + 设备管理 + 人员管理 + 销售管理（受 DataScope 约束） | 日常管理 |
| `manager` | 部门主管 | 本部门及下级数据查看（DataScope=3），有限的操作权限 | 部门管理 |
| `equipment_admin` | 设备管理员 | 设备管理模块（装置/设备/传感器/摄像头/维保/报警配置） | 设备运维 |
| `yuangong` / `employee` | 普通员工 | 仅首页 + 个人中心，只读权限 | — |

## 5. 核心业务流程

### 5.1 用户创建 → 角色分配 → 权限生效（系统管理核心链路）

```
1. 管理员登录 → 侧边栏「系统管理」→「用户管理」(/system/user)
       │
2. 点击「新增」→ 弹出 Dialog
   填写: username(2-20字符,唯一) / name / password / confirmPassword
        phone(11位) / email / deptId(部门树选择) / roleIds(角色多选)
       │
3. 前端校验 → 点击「确定」→ POST /api/system/user
       ├── 成功 → Toast "新增成功" → 关闭弹窗 → 刷新列表
       └── 失败 → Toast/表单错误提示（如"用户名已存在"）
       │
4. 分配角色（两种方式）:
   ├── 方式A: 新增/编辑时在表单中选择 roleIds
   └── 方式B: 列表行点击「分配角色」→ 弹出 Checkbox Dialog → 勾选角色
            → PUT /api/system/user/role → Toast → 刷新列表
       │
5. 角色关联菜单:
   [角色管理] 点击「分配菜单」→ 树形 Checkbox 选择菜单项
            → PUT /api/system/role/menu → 权限生效
       │
6. 权限生效验证:
   ├── 侧边栏: 动态路由按 role 过滤 → 仅渲染有权限的菜单项

```

```
   │
   ├─ 1. 新增合同（草稿状态）
   │   POST /api/sales/contract
   │   填写: 合同编号 / 客户(Select) / 产品 / 合同金额 / 生效日期
   │   └── 初始状态: PENDING（待执行）
   │
   ├─ 2. 启动合同
   │   PUT /api/sales/contract/{id}/start
   │   └── 状态: EXECUTING（执行中）
   │   此时可创建关联销售订单，订单金额累计影响"未执行额"
   │
   ├─ 3a. 正常终止
   │   PUT /api/sales/contract/{id}/terminate
   │   └── 状态: TERMINATED（已终止）
   │   已终止的合同不允许编辑、不允许删除
   │
   └─ 3b. 正常完成
       当所有关联订单执行完毕、未执行额归零
       └── 状态: COMPLETED（已完成）
       进度条显示 100%

关键校验:
  - 已终止/已完成的合同不可编辑（UI按钮置灰 + 后端校验）
  - 待执行合同可删除；已启动后的合同不可删除
  - 合同金额必须 > 实际执行金额
```

| 1 | **侧边栏导航 356 ERROR** | SPA 路由切换时侧边栏等待超时不够(5s)；Element Plus el-menu CSS 选择器不稳定 | JS `location.hash` 直接操作路由 + `_navigate_by_js_hash()` 作为首选导航方式 + `_wait_page_content_ready()` 表格列数稳定轮询（6次重试） | `base/sidebar_navigator.py` |

### 9.2 经验教训

| # | 教训 | 建议 |
|:---:|------|------|
| 1 | **渲染时序是最大失败根因 (40%)** — 测试比页面快，表头/分页/列表渲染延迟 | 所有页面操作前：`wait_page_content_ready()` + 8s 全局等待；表格操作后：`wait_vue_stable()` |
| 2 | **定位器不要耦合 Element Plus 内部 class** — `el-button--primary` / `el-table__row` 等不稳定 | 使用 JS 文本搜索（最稳定）→ XPath（次选）→ CSS（最后选择）；推动开发添加 `data-testid` |
| 3 | **种子数据必须预置** — RBAC 测试完全依赖预置的角色/用户/菜单 | 每个模块 conftest.py 添加 `ensure_seed_data` fixture；seed 脚本独立可运行 |
| 4 | **API 变更需同步更新 Page Object** — 合同管理 10条用例失败的根因包含 API 脱节 | 建立 API 变更通知机制；Page Object 字段映射使用配置文件而非硬编码 |
| 5 | **页面加载超时配置应集中管理** — 多处硬编码 30s | 统一从 `config/base.py` 读取 `page_load_timeout`(60s) |
| 6 | **30 个路由全在线是好信号** — 失败全是定位器+时序问题，没有"页面不存在" | 集中火力修复定位器层和时序层，不需要排查后端路由 |
